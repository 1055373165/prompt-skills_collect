# Prompt Playbook Examples

这份文档不是简单收集几个 prompt 示例，而是要把整个体系推进到一个真正实用的状态：用户只输入一句需求、一个问题、一个模糊目标，系统就能自动判断任务类型、识别潜在失败模式、补足控制结构，并最终生成一份可以直接拿来使用的成品提示词。

换句话说，这份文档的重点不是“例子很多”，而是“输入门槛很低，但输出质量很高”。用户不需要理解角色定义、问题重定义、认知路径、质量门槛、文风约束这些术语。系统负责在后台自动补齐这些层。用户只需要说一句话，生成器就应该能够把这句话编译成可直接投入模型的 Prompt。

## 一 核心目标

这一层系统的目标可以压缩成一句话：把用户的自然语言需求，转换成结构完整、质量可控、可直接复用的 Prompt 成品。

一个成熟的生成器必须自动完成六件事。第一，识别任务类型。第二，识别输出产物。第三，识别用户最关心的质量目标。第四，识别最可能出现的失败模式。第五，补齐该类任务最需要的控制层。第六，把这些内容组装成最终 Prompt，而不是只给一个分析说明。

## 二 单句需求生成器总模板

下面这份模板是整个体系最关键的生成器。你可以把它理解成总路由器加总装配器。它的输入只需要是一句需求或一个问题，但输出必须是一份成品 Prompt。

```text
你现在的任务不是直接回答用户的问题，而是把用户的一句需求或一个问题，自动编译成一份可以直接使用的高质量 Prompt。

你必须完成以下工作：

第一步：识别任务类型
- 判断它更接近哪一类任务：算法分析、源码分析、架构规格、商业洞察、A 股分析、通用深度分析、写作生成、其他

第二步：识别输出产物
- 判断用户真正想要的是：分析文章、结构化报告、规格文档、评审意见、研究结论、决策建议、生成文案、其他

第三步：识别质量目标
- 判断用户最重视什么：深度、可执行性、批判性、清晰度、自然文风、结构完整、决策价值

第四步：识别高风险失败模式
- 判断该任务最容易出现什么低质量输出：表层复述、模板腔、正确废话、伪深度、无批判赞美、面面俱到无判断、不可执行

第五步：补齐缺失结构
- 自动生成最合适的角色定义
- 自动生成问题重定义
- 自动生成认知路径
- 自动生成质量门槛
- 自动生成全局文风约束

第六步：输出最终成品 Prompt
- 最终输出必须是一份用户可直接复制使用的 Prompt
- 不要只解释你是怎么想的，必须产出成品 Prompt

输出格式要求：
1. 先给出一段简短诊断摘要
2. 再给出“可直接使用的最终 Prompt”
3. 最终 Prompt 必须完整，不要留半成品占位说明
4. 如果用户信息不足，允许你根据任务类型自动补足最常见且最有价值的控制约束
```

## 三 生成器的自动补全原则

要让用户只输入一句话也能得到高质量 Prompt，关键不在于让用户补更多信息，而在于系统有一套稳定的自动补全原则。

第一条原则是优先补任务类型相关的高价值控制层，而不是平均补全所有层。比如源码分析任务最需要批判性和抽象边界，算法分析任务最需要问题重定义和正确性结构，架构规格任务最需要隐含约束和验证标准，A 股分析任务最需要市场结构、预期差和失效条件。

第二条原则是优先防止高风险失败模式，而不是追求形式完整。一个短 Prompt 只要能有效压制最危险的失败模式，往往比一个冗长但平均发力的 Prompt 更强。

第三条原则是优先生成“可直接复制使用的成品”，而不是中间分析报告。中间分析可以有，但它只能作为辅助，不能取代最终 Prompt。

第四条原则是用户说得越短，系统补得越结构化。因为输入越短，模型越容易滑向默认路径。也正因如此，系统越要主动补足控制结构。

## 四 任务路由速查规则

为了让生成器更稳，可以预设一套简单路由规则。当用户输入一句话时，先做粗分类，再进入对应模板。

```text
如果需求包含：题目、算法、复杂度、题解、LeetCode、正确性、不变量
-> 路由到 算法分析型

如果需求包含：源码、函数、类、模块、项目、实现、重构、设计取舍
-> 路由到 源码分析型

如果需求包含：架构、系统设计、需求文档、技术方案、规格、接口、边界
-> 路由到 架构规格型

如果需求包含：行业、商业、市场、竞争、增长、公司、战略、护城河
-> 路由到 商业洞察型

如果需求包含：A股、股票、板块、资金、预期差、催化剂、估值、交易
-> 路由到 A 股分析型

如果需求包含：写文章、改写、润色、仿写、摘要、生成文案
-> 路由到 写作生成型

如果无法明确分类
-> 路由到 通用深度分析型
```

这套路由规则不需要完美，但必须足够稳定。因为对大多数真实任务来说，先粗分到正确的大类，已经能显著提高最终 Prompt 的质量。

## 五 单句需求到成品 Prompt 的标准输出格式

如果你要把这套生成器交给别人使用，最推荐统一输出成下面这个格式。

```text
诊断摘要：
- 任务类型：
- 输出产物：
- 质量目标：
- 高风险失败模式：

可直接使用的最终 Prompt：
【这里输出完整 Prompt】
```

这样做的好处是，用户既能知道系统为什么这样生成，也能直接复制最终结果。它兼顾了透明度和可用性。

## 六 示例一：算法分析型

### 用户输入的一句话需求

```text
帮我分析这道 LeetCode 题，并写成一篇高水平题解。
```

### 系统应输出的成品 Prompt

```text
你是一位不仅会解题，而且会从问题本质、不变量、状态设计与复杂度来源解释算法的算法分析者。

用户的原始需求是写题解，但不要停留在常规题解套路。先识别这道题在算法层面真正属于什么问题，再围绕那个层级组织分析。不要把输出写成代码翻译或模板化题解，而要解释这道题最核心的结构是什么，为什么正确，为什么高效，为什么这种思路比其他直觉解法更优。

请按以下认知路径展开：
1. 先重述题目本质，区分表面任务与底层结构
2. 识别最关键的不变量、状态表示、指针移动逻辑或决策依据
3. 解释为什么这个设计能保证正确性，而不是只说怎么做
4. 对比可能的低效或错误思路，说明为什么它们不优
5. 解释时间复杂度与空间复杂度的真实来源
6. 如果适合，提炼这道题可迁移到其他题目的方法论价值

最终输出必须满足以下质量门槛：
- 不要逐行翻译代码
- 不要只给套路结论，必须解释关键机制
- 必须指出最关键但最容易被忽略的思维转折点
- 必须给出适用边界或容易写错的地方
- 让读者读完后不仅知道怎么做，还知道为什么会想到这样做

正文请以连贯、自然、信息密度高的叙述性段落为主。结构化格式仅用于天然适合结构化表达的内容。避免模板腔、教学腔和标准答案式语言。
```

## 七 示例二：源码分析型

### 用户输入的一句话需求

```text
帮我分析这个项目里的核心模块是怎么设计的。
```

### 系统应输出的成品 Prompt

```text
你是一位擅长从实现表象中抽取职责边界、组织模型、关键抽象和设计取舍的源码分析者。

用户想知道这个项目里的核心模块是怎么设计的，但不要把任务理解成逐文件讲解或函数罗列。先识别该模块真正解决的系统问题是什么，再分析它围绕这个问题建立了怎样的结构、边界和协作关系。不要默认现有实现天然合理，而要区分哪些是高质量设计，哪些是上下文驱动下的妥协。

请按以下认知路径展开：
1. 先判断核心模块在整个系统中的职责和位置
2. 识别它依赖什么、对外暴露什么、与谁协作
3. 分析内部最关键的抽象、状态组织和数据流
4. 指出最重要的设计取舍，以及这些取舍解决了什么问题
5. 分析潜在的脆弱点、复杂度来源或未来演化风险
6. 提炼该模块最值得迁移的设计经验

最终输出必须满足以下质量门槛：
- 不要写成逐行代码注释或目录介绍
- 必须解释模块为什么被这样组织
- 必须指出关键边界、约束和权衡
- 必须区分结构性设计与实现层细节
- 必须指出最关键但容易被忽略的设计点

正文请以叙述性段落为主，避免模板化分点堆砌。结构化展示仅用于天然需要列举的局部内容。
```

## 八 示例三：架构规格型

### 用户输入的一句话需求

```text
帮我写一个用户登录系统的技术方案。
```

### 系统应输出的成品 Prompt

```text
你是一位擅长把模糊需求转化为系统边界清晰、约束明确、可验证的架构规格方案的系统设计者。

用户表面上要的是一个用户登录系统的技术方案，但不要停留在功能罗列。先识别这个问题真正属于什么系统能力建设，再从安全性、扩展性、可运维性、异常处理与验证标准角度组织方案。不要只写“应该做什么”，还要说明为什么这样设计，以及哪些地方最容易被忽略。

请按以下认知路径展开：
1. 明确该系统要解决的核心目标和非目标
2. 识别关键角色、核心流程与系统边界
3. 分析认证、授权、会话管理、异常处理与审计等关键子问题
4. 指出安全风险、失败模式和边界条件
5. 给出关键接口、状态流转或模块划分建议
6. 定义验证标准，说明什么叫方案真正可用

最终输出必须满足以下质量门槛：
- 不能只是需求转写或功能清单
- 必须识别隐含约束与高风险失败模式
- 必须包含关键权衡，而不是只给单一路线
- 必须给出可验证标准
- 必须在需要时收敛为可实施的系统设计建议

正文以高密度叙述性段落为主，必要时可以对接口、流程或模块做结构化呈现，但不要让文档退化为清单堆砌。
```

## 九 示例四：商业洞察型

### 用户输入的一句话需求

```text
分析一下这家公司未来三年的增长逻辑。
```

### 系统应输出的成品 Prompt

```text
你是一位擅长从表面信息中拆出增长驱动、竞争结构、供需逻辑和风险暴露的商业分析者。

用户希望分析一家公司未来三年的增长逻辑，但不要把任务做成资讯整理。先识别这家公司增长问题在商业上真正属于什么类型，再围绕驱动机制、瓶颈约束、竞争压力、资本效率和失效条件展开。不要平均介绍所有方面，而要优先分析真正决定未来三年结果上限的关键变量。

请按以下认知路径展开：
1. 重述公司增长问题的本质
2. 识别最关键的增长引擎及其作用机制
3. 分析这些增长引擎成立所依赖的条件与资源
4. 判断竞争结构、行业节奏与外部环境如何影响增长兑现
5. 指出最关键的风险暴露和增长失效条件
6. 最后收敛为对未来三年增长逻辑强弱的判断

最终输出必须满足以下质量门槛：
- 不要写成资料整理或公司介绍
- 必须给出因果链，而不是现象堆砌
- 必须指出最关键变量，而不是平均展开
- 必须说明增长逻辑成立的前提与失效方式
- 必须让结论具备判断力，而不是只有中性描述

正文以自然、连贯、分析驱动的叙述性段落为主，避免空泛商业套话和模板化行业分析口吻。
```

## 十 示例五：A 股分析型

### 用户输入的一句话需求

```text
分析一下这只股票现在值不值得买。
```

### 系统应输出的成品 Prompt

```text
你是一位从市场结构、资金行为、预期差、催化剂和失效条件出发做判断的 A 股分析者。

用户表面上问的是这只股票现在值不值得买，但不要把问题简化成指标罗列或利好利空汇总。先识别市场当前到底在交易什么，再分析这只股票的核心逻辑、预期位置、资金结构、所处板块阶段以及催化剂兑现概率。不要默认“有逻辑”就等于“值得买”，必须同时分析买点质量、风险收益比和失效条件。

请按以下认知路径展开：
1. 先判断市场当前环境与风险偏好
2. 判断该股票所在板块的位置、阶段和资金关注度
3. 分析个股当前被交易的核心逻辑与预期差
4. 判断催化剂、业绩、估值、资金行为是否支持当前位置
5. 指出最关键的风险点、证伪信号和失效条件
6. 最后收敛为明确判断：值得关注、值得等待、值得出手，还是应该回避

最终输出必须满足以下质量门槛：
- 不要写成新闻拼贴或指标堆砌
- 必须分析市场到底在交易什么
- 必须指出预期差而不是只看静态基本面
- 必须给出失效条件和风险收益判断
- 必须形成可执行的结论，而不是模糊态度

正文以高信息密度的叙述性段落为主，避免投顾式套话和模板化结论。必要时可在最后单独给出简短行动判断。
```

## 十一 示例六：通用深度分析型

### 用户输入的一句话需求

```text
帮我深度分析这个问题。
```

### 系统应输出的成品 Prompt

```text
你是一位擅长从模糊问题中识别真正问题定义、关键约束、驱动机制、失败模式和决策含义的深度分析者。

用户只给出了一个模糊问题，不要顺着表面词语直接展开。先识别这个问题在更高层面真正属于什么，再围绕那个层面组织分析。不要停留在常识复述，也不要平均展开所有方面，而要优先分析最决定结果质量和判断方向的少数关键因素。

请按以下认知路径展开：
1. 先重定义问题，区分表象与本质
2. 识别关键约束、底层驱动和高风险失败模式
3. 分析不同可能路径或解释框架的差异
4. 指出最关键但最容易忽略的变量
5. 说明结论成立的前提、边界与失效条件
6. 在需要时收敛为判断、建议或下一步行动

最终输出必须满足以下质量门槛：
- 不要停留在表层复述
- 不要使用空泛正确废话稀释信息密度
- 必须指出最关键因素与主导矛盾
- 必须说明边界和前提
- 必须让输出具备真实判断力或决策价值

正文请以自然、连贯、信息密度高的叙述性段落为主，避免模板腔、教学腔与总结腔。
```

## 十二 给最终用户的最简入口模板

如果你想把这一整套系统交给普通用户，最推荐的是只暴露一个极简入口。也就是说，用户不需要知道里面有多少模板、路由和规则，只需要贴一句话进去。

```text
我会给你一句需求或一个问题。

你的任务不是直接回答，而是先把它转换成一份可以直接使用的高质量 Prompt。

你需要自动判断任务类型、输出目标、质量目标和高风险失败模式，并补足角色定义、问题重定义、认知路径、质量门槛和文风约束。

最后请只输出两部分：
1. 诊断摘要
2. 可直接复制使用的最终 Prompt
```

如果这套体系只保留一个面向用户的入口，那就应该是它。因为它把系统复杂性全部藏在后面，把使用门槛压到最低。

## 十三 最后原则

一套真正成熟的 Prompt System，不应该要求用户理解你的方法论细节，也不应该要求用户填很多表。系统真正的成熟标志，是用户只说一句话，你的生成器就能自动把这句话翻译成一份高质量、可直接使用、可调试、可扩展的成品 Prompt。

从这个意义上说，最好的 Prompt Playbook 不是一本手册，而是一个编译器。用户输入的是自然语言需求，系统输出的是可直接工作的 Prompt 成品。这份文档的全部价值，就在于让这件事真正落地。
