# Prompt Engineering Handbook v2

如果说 v1 解决的是提示词工程的基本认知框架，那么 v2 的重点就是把这套框架推进到可实验、可调试、可自动组装的层面。很多人对提示词的理解停留在经验主义阶段，也就是记住一些有效句式，遇到任务时凭感觉拼接，然后从结果中主观判断是否更好。这个阶段虽然能解决部分问题，但很快就会撞到上限，因为它无法稳定复现，也无法解释为什么某条提示在某类任务中有效，而换一个任务就效果平平。要继续深入，就必须把提示词工程从写作技巧转化成一种更接近实验科学和控制工程的实践。

更完整的提示词工程，不再把 prompt 看成一段静态文本，而是把它看成一个由多个控制变量组成的系统输入。系统输出的质量，不取决于单句是否华丽，而取决于变量之间是否形成了清晰的分工与有效的协同。角色定义决定模型以什么判断框架进入问题，问题重定义决定模型到底在解哪一个问题，认知路径决定它必须经过哪些思维台阶，表达约束决定它会沿着哪种文风轨迹展开，质量门槛决定它在什么条件下才允许“停机”。一旦你开始按这种方式理解 prompt，你就不再是在寻找万能模板，而是在做控制策略设计。

## 六 实验方法论

如果想真正掌握提示词工程，最值得建立的不是一个更大的模板库，而是一套自己的实验方法。因为提示词最容易让人误入的陷阱，就是把偶然有效当成普遍规律。你在一个任务上加了一句角色设定，效果明显变好，于是就认为角色定义永远最重要。但很多时候它改善的只是语气与判断姿态，而不是深度本身。你加入一条叙述性写作约束，AI 味显著下降，于是又认为风格控制最关键。但它可能同时降低了扫读性或结构清晰度。只有进入实验范式，你才能把这些变化看清楚。

最推荐的实验方法是单变量干预。也就是说，在同一个任务、同一份输入和尽量相似的输出要求下，每次只改变一个提示变量，观察它对结果的具体影响。你不需要做学术级别的严格实验，但至少要做到控制变量的意识。例如先用基础 prompt 得到一个基线输出，然后只加入角色定义，再对比它对判断密度、问题重定义能力、术语选择、文风姿态的影响。然后在基线 prompt 上只加入认知路径约束，观察它是否提升了结构完整性和分析深度。再只加入文风约束，观察它对模板腔、分点腔和自然度的影响。这样几轮之后，你开始得到的就不是印象，而是经验中带证据的判断。

一个成熟的实验记录，至少应包含五个维度。任务类型说明你在分析什么场景，原始 prompt 说明系统输入是什么，输出缺陷说明当前答案的问题在哪里，干预变量说明你增加了什么控制约束，修复效果说明它到底改善了什么、又引入了什么副作用。如果你长期记录这些信息，最后会发现自己积累的不再是 prompt 样例，而是一套关于提示控制变量的行为数据库。

下面是一个适合长期使用的实验记录模板。

```text
任务名称：
任务类型：
输入内容：

基线 Prompt：

基线输出缺陷：
- 
- 

本轮新增控制变量：

观察到的改善：
- 
- 

观察到的新问题：
- 
- 

结论：
这条控制变量更适合解决什么问题，不适合解决什么问题。
```

当你开始以这种方式工作后，提示词工程会发生一个质变。你不再是在“感觉上优化 prompt”，而是在建立一套关于模型行为的经验科学。真正厉害的不是谁记住更多漂亮句子，而是谁更清楚某个变量在什么任务中起主作用，在什么任务中只是辅助修饰。

## 七 从需求到成品 Prompt 的自动组装流程

一套真正成熟的提示词系统，不应该要求用户每次手工写出高质量 prompt。更合理的方向，是把提示词拆成中间层变量，让系统根据用户需求自动生成更完整的主提示词。这个思路的关键，不是自动补全语言，而是自动补全思维控制结构。也就是说，用户只描述任务，系统负责推断它属于什么问题类型，需要什么角色定义，最可能掉进哪些失败模式，因此应该补哪些约束。

一个可行的自动组装流程可以分成五步。第一步是识别任务对象，也就是用户到底在处理源码、算法、架构、商业、股票分析、文案写作还是其他主题。第二步是识别任务产物，也就是用户到底想得到文章、规格文档、评审意见、研究报告还是行动建议。第三步是识别质量目标，也就是用户更强调深度、可执行性、批判性、自然文风还是结构完整。第四步是识别高风险失败模式，也就是这个任务最容易滑向表层复述、模板腔、不可执行、无批判赞美还是伪深度。第五步才是根据以上推断，自动组装角色定义、认知路径、质量门槛和文风约束。

如果把这个流程压缩成一个可执行框架，它看起来会像这样。

```text
输入：用户原始需求

步骤一：识别任务类型
步骤二：识别输出产物
步骤三：识别用户最重视的质量目标
步骤四：识别该任务的高风险失败模式
步骤五：生成最适合的角色定义
步骤六：生成该类任务所需的认知路径
步骤七：补充对应的质量门槛
步骤八：附加全局写作约束

输出：一份由角色层、问题重定义层、认知动作层、质量门槛层、全局文风层组成的完整 Prompt
```

这种自动化的意义非常大，因为它把提示词生成从文本创作问题，转化成任务诊断问题。用户不再需要懂专业术语，也不需要知道什么叫不变量、抽象边界或失败模式。他只需要描述自己要完成什么，系统就能替他补足真正决定结果质量的控制结构。这正是把提示词工程从手工艺推向半自动工程化的关键一步。

## 八 任务配方库

随着你对不同任务的理解加深，你会发现并不是所有任务都需要同样的提示骨架。有的任务最需要问题重定义，有的最需要批判性，有的最需要可执行性，还有的最需要风格控制。为了提高复用效率，最好的方法是沉淀一套任务配方。配方不是完整 prompt，而是针对某类任务，明确它最核心的控制层应如何组合。

源码分析类任务最值得强化的是角色定义、问题重定义和批判性指令。因为这类任务最容易滑向逐行讲解、函数罗列和无条件赞美。算法分析类任务最值得强化的是问题重定义、认知下钻和边界验证。因为它最容易滑向套路题解和逐行代码翻译。架构规格类任务最值得强化的是隐含需求识别、失败模式分析和可验证标准。因为它最容易退化为需求转写和功能列表。商业洞察类任务最值得强化的是因果结构判断、叙事去噪和决策收敛。因为它最容易变成资讯整理。A 股分析类任务最值得强化的是市场结构感知、预期差判断、催化剂与失效条件分析。因为它最容易滑向指标堆砌和新闻拼贴。

为了让你后续能更快拼装，这里给出一组简化配方。

```text
源码分析配方 = 角色定义 + 问题重定义 + 批判性指令 + 迁移价值 + 全局文风层

算法分析配方 = 问题重定义 + 不变量/状态设计 + 正确性条件 + 复杂度解释 + 全局文风层

架构规格配方 = 角色定义 + 隐含需求识别 + 系统边界 + 失败模式 + 可验证标准 + 全局文风层

商业洞察配方 = 问题重定义 + 因果链分析 + 竞争结构 + 风险暴露 + 决策含义 + 全局文风层

A 股分析配方 = 市场阶段判断 + 资金与预期分析 + 板块位置 + 催化剂/失效条件 + 决策收敛 + 全局文风层
```

这些配方的真正作用，不是让你机械复用，而是让你在面对新任务时，先问自己这类任务的主失败模式是什么，再决定优先放大哪一组控制变量。高手和普通用户的区别，经常不在于他写了更多内容，而在于他更快识别了该任务真正缺的是哪一层约束。

## 九 Prompt Debugging 决策树

提示词工程进入高阶阶段后，你最常做的事情不会是从零写 prompt，而是调试一个已经能工作但质量还不够高的 prompt。这时最有用的，不是更多模板，而是一棵决策树。它的目的，是在你拿到一个不理想输出时，快速判断问题属于哪一类，再补哪一层约束。

如果输出很相关但没有洞察，先检查是否缺少问题重定义与认知下钻。因为模型很可能只是在复述输入。若输出结构清晰但 AI 味很重，先检查是否缺少全局文风约束。若输出有一定深度但不成体系，说明认知路径可能不完整，模型缺少展开分析的台阶。若输出显得很专业却无法落地，说明缺少可验证标准和决策收敛。若输出几乎都在肯定现有方案，则要补批判性指令和权衡分析要求。若输出覆盖很全却没有重点，则应加入关键点优先和信息密度约束。

下面给出一个可直接复用的调试决策树简表。

```text
如果输出像复述 -> 补 问题重定义 + 认知下钻
如果输出像模板 -> 补 全局文风约束 + 禁止模板引导语
如果输出正确但空 -> 补 信息密度约束 + 关键点优先
如果输出高深但虚 -> 补 边界/验证约束 + 可执行性要求
如果输出只会赞美 -> 补 批判性指令 + 代价/局限分析
如果输出很全但不尖锐 -> 补 关键点优先 + 主问题聚焦
如果输出有洞察但难以行动 -> 补 决策收敛 + 成功标准 + 触发条件
如果输出局部很好整体不稳 -> 补 独立全局文风层 + 输出节奏约束
```

这棵决策树的价值，在于它把 prompt 优化从文学修改转化成了故障排查。你不再因为结果不好就盲目换词，而是先判断它属于哪一类偏差，再做最小必要修复。长期这样训练，你会形成一种非常重要的能力，也就是从输出问题直接反推出提示层缺了什么结构。

## 十 提示词的最小完备结构

如果要把前面所有内容进一步压缩成一个最实用的结论，那么一份高质量 prompt 至少应包含五层。第一层是角色与判断框架，用来激活正确的能力簇。第二层是问题重定义，用来防止模型停留在用户原话表面。第三层是认知动作，用来让模型必须经过关键思维步骤。第四层是质量门槛，用来定义什么叫真正完成。第五层是全局文风约束，用来把输出从高频模板路径中拉出来。

这五层并不意味着每次都要写得很长。真正熟练之后，你会发现很多高质量 prompt 可以非常短，但它的每一句都在承担一层控制职责。反过来说，一个看起来很长的 prompt，如果没有这五层中的关键部分，也仍然可能是低质量 prompt。长度从来不是本质，控制结构才是本质。

你甚至可以把这五层压缩成一个极简骨架，用作任何任务的起始模板。

```text
你是一位【具备特定判断方式的角色】。
用户的描述是起点，不是边界。先识别问题真正属于什么，再组织回答。
不要停留在表层解释，必须分析关键约束、代价、失败模式与边界条件。
最终输出必须解释为什么，指出最关键因素，给出适用边界，并在需要时收敛为可执行判断。
正文以叙述性段落为主，结构化格式仅用于天然适合结构化表达的内容。
```

当你能熟练地在不同任务中变形这五层结构时，你基本就已经跨过了初级提示词工程阶段。后面的提升，更多来自实验积累、失败模式辨识能力和元指令库的成熟度。

## 十一 未来迭代方向

如果这份手册要继续升级，最值得增加的不是更多模板，而是更多可操作的实验与诊断工具。比如建立任务类型与失败模式之间的映射表，形成不同任务的最佳起手 prompt 配方，再进一步把需求到 prompt 的自动组装做成半结构化流程。你还可以逐步沉淀一个自己的元指令词典，把那些在多类任务中都稳定有效的句子单独提取出来，像积木一样复用。

长期来看，真正让你和大多数用户拉开差距的，不会是你记住了多少条提示词，而是你是否形成了三种底层能力。第一种是把一个模糊任务重新定义成模型更容易高质量求解的问题。第二种是把一个差输出诊断成某类失败模式，并做最小必要修复。第三种是把一次次有效实践抽象成可组合、可迁移的控制语言。一旦这三点建立起来，提示词工程对你来说就不再是经验技巧，而会变成一套真正可持续演化的方法论.
