# Prompt System Playbook

这不是一份介绍 prompt engineering 的说明文，而是一份可以直接投入日常工作的操作手册。它的目标不是让你理解一些抽象概念后自行发挥，而是让你在拿到真实任务时，能按固定步骤完成任务诊断、Prompt 组装、输出调试、质量评估和经验沉淀。你应当把它理解成一套可重复执行的工作系统，而不是一篇理论文章。

一份真正可用的 Prompt System Playbook，至少要解决六个问题。第一，拿到任务时，如何快速判断这个任务到底属于什么。第二，如何从一个模糊需求稳定生成一份结构完整的高质量 Prompt。第三，如何避免模型滑向复述、模板腔、正确废话和伪深度。第四，如何在输出不理想时进行最小必要修复，而不是推倒重来。第五，如何评估一个 prompt 到底好不好，而不是凭感觉。第六，如何把一次次有效实践沉淀成自己的可复用系统。

如果你愿意把这份手册真正用起来，那么你以后做的大多数复杂任务，都不应该再从“请帮我分析一下”开始，而应该从一次任务诊断开始。诊断之后，再做 Prompt 组装；组装之后，再做执行观察；观察之后，再做调试修复；修复之后，再做沉淀记录。这个闭环一旦建立，prompt 就不再是一次性文本，而会变成你手中的系统工件。

## 一 使用方式

这份 Playbook 可以按三种方式使用。

第一种方式，是把它当成日常 SOP。每次拿到复杂任务，就按下面的标准流程走一遍。第二种方式，是把其中的模板块复制出来直接用，适合你已经熟悉方法、只想快速落地的时候。第三种方式，是把它当成团队内部的统一协议，确保不同人写出来的 prompt 至少具备同样的结构完整性和可调试性。

如果你是第一次使用，建议从这一顺序开始。先用“任务输入卡”收集信息，再用“Prompt 组装器”生成第一版 prompt，再用“输出评估表”判断结果质量，最后在结果不佳时使用“Prompt 调试器”。只要按这个顺序执行几轮，你会很快形成稳定手感。

## 二 标准工作流

下面是一套适合大多数复杂任务的标准工作流。除非任务只是简单事实查询，否则都建议按这套流程推进。

```text
步骤一：填写任务输入卡
步骤二：完成任务诊断
步骤三：生成第一版 Prompt
步骤四：执行并获取第一版输出
步骤五：使用输出评估表判断缺陷
步骤六：使用 Prompt 调试器做最小必要修复
步骤七：重新执行并比较前后结果
步骤八：把有效约束沉淀到元指令库
```

这条流程看似比直接写 prompt 更慢，但真实情况通常相反。因为大多数时间浪费，不是浪费在构造 prompt 上，而是浪费在低质量输出之后的反复返工上。工作流的价值，就是把返工从盲改变成有方向的修复。

## 三 任务输入卡

任何高质量 Prompt 的起点，都不应该是自由发挥，而应该是一次结构化输入。下面这张输入卡可以直接复制使用。它是整个 Playbook 的入口。

```text
任务输入卡

1. 你在处理什么内容：
2. 你希望最终得到什么产物：
3. 这个产物主要服务于什么：理解 / 决策 / 执行 / 表达
4. 你最看重的质量目标是什么：深度 / 可执行性 / 批判性 / 清晰度 / 自然文风 / 结构完整
5. 你最不希望输出出现什么问题：复述 / 模板腔 / 空话 / 伪深度 / 只会赞美 / 不可执行
6. 你的输入材料是什么：源码 / 题目 / 需求 / 文档 / 数据 / 观点 / 新闻 / 混合材料
7. 是否有明确边界或约束：
8. 是否需要最终收敛为结论、建议、规格或行动清单：
```

如果你只有极少信息，也至少应填写前五项。因为这五项已经足够让系统完成任务类型判断、质量目标判断和失败模式预判。

## 四 任务诊断器

拿到输入卡之后，不要立刻写 prompt。先做一次诊断。诊断的意义，在于把用户口头需求转化成 prompt 设计所需的控制信息。

下面是可直接复制的诊断模板。

```text
请先不要直接完成任务，而是先完成任务诊断。

基于我提供的信息，输出以下内容：

1. 任务类型
2. 输出产物类型
3. 这个任务真正要解决的问题是什么
4. 这个任务最关键的质量目标是什么
5. 这个任务最可能出现的三个失败模式是什么
6. 最应该强化的三个 Prompt 控制层是什么
7. 最适合的角色定义方向是什么

最后再给出一句简洁判断：这类任务最怕缺什么。
```

你使用这份诊断器时，不是为了得到一段漂亮分析，而是为了得到 Prompt 组装的依据。也就是说，诊断结果应该直接决定下一步 Prompt 长什么样，而不是停留在抽象层面。

## 五 Prompt 组装器

完成诊断后，再进入 Prompt 组装。组装不是写一大段看起来很完整的话，而是明确拼出五层结构：角色层、问题重定义层、认知路径层、质量门槛层、全局文风层。下面这个组装器可以直接复制使用。

```text
你现在不是直接完成任务，而是根据用户需求为这个任务组装一份高质量 Prompt。

请按以下顺序完成：

第一部分：任务诊断摘要
- 判断任务类型
- 判断输出产物
- 判断最重要的质量目标
- 判断最危险的失败模式

第二部分：生成 Prompt 的五层结构

1. 角色层
- 角色必须体现判断方式，而不是只有头衔

2. 问题重定义层
- 把用户描述从表层任务重写成结构性问题

3. 认知路径层
- 明确模型在输出前必须完成哪些分析动作

4. 质量门槛层
- 明确什么样的答案才算真正完成

5. 全局文风层
- 明确输出风格、结构化边界和模板化禁区

第三部分：输出最终可直接使用的完整 Prompt

要求：
1. Prompt 必须由上面五层构成
2. 每一层都要服务于结果质量，而不是形式完整
3. 如果需求模糊，优先补足最能防止低质量输出的约束
4. 先给诊断摘要，再给最终 Prompt
```

这是整个 Playbook 的核心模板。你可以把它当作 prompt compiler。你输入需求，它输出结构化 prompt，而不是直接输出成品答案。

## 六 Prompt 最小完备骨架

如果你不想每次都用完整组装器，那么至少可以从下面这个最小骨架开始。它不是最强版本，但足够避免大多数低质量输出。

```text
你是一位【具备特定判断方式的角色】。
用户的描述是起点，不是边界。先识别问题真正属于什么，再组织回答。
不要停留在表层解释，必须分析关键约束、代价、失败模式与边界条件。
最终输出必须解释为什么，指出最关键因素，给出适用边界，并在需要时收敛为可执行判断。
正文以自然、连贯的叙述性段落为主，结构化格式仅用于天然适合结构化表达的部分。
```

如果你的工作环境很快、任务很多、来不及做完整诊断，那么用这个最小骨架作为起手式，已经会比绝大多数随手 Prompt 稳定得多。

## 七 日常高频模板

下面这些模板不是为了展示理论，而是为了让你直接复制进工作流里使用。

### 1. 需求转 Prompt 模板

```text
我会给你一个原始需求。

你的任务不是直接完成需求，而是先把它升级成一份高质量 Prompt。

请完成以下工作：
1. 识别任务类型
2. 识别输出产物
3. 识别用户最在意的质量目标
4. 识别最可能出现的失败模式
5. 基于以上信息生成一份完整 Prompt

Prompt 必须包含：角色定义、问题重定义、认知路径、质量门槛、文风约束。

最后请单独输出“最终 Prompt”，以便我直接复制使用。
```

### 2. 先诊断后回答模板

```text
请不要直接回答问题。

先完成以下诊断：
1. 这个问题表面上在问什么
2. 它本质上属于什么问题
3. 如果只顺着表面回答，最容易产生什么低质量结果
4. 真正高质量的回答应该重点解决什么

完成诊断后，再输出最终回答。
```

### 3. 提升深度模板

```text
不要停留在表层解释或常识复述。
必须继续分析：
1. 问题真正的底层结构是什么
2. 哪些因素决定结果上限
3. 哪些看起来次要、实际上很关键
4. 这个判断成立的前提和边界是什么
```

### 4. 提升可执行性模板

```text
不要把结论停留在分析层。
最终输出必须收敛为可执行判断，明确：
1. 下一步该做什么
2. 在什么条件下做
3. 成功标准是什么
4. 出现什么信号时应调整或停止
```

### 5. 降低 AI 味模板

```text
正文请以自然、流畅、连贯的叙述性段落为主。
不要使用高频教学腔、总结腔和模板引导语。
不要把正文写成分点堆砌。
结构化格式只用于天然适合结构化表达的内容。
```

## 八 Prompt 调试器

高质量工作不是从零重写 prompt，而是修 prompt。你需要一个专门用于调试的工具模板。下面这份可以直接用于 bad output 修复。

```text
下面我会给你三个内容：
1. 原始任务
2. 当前 Prompt
3. 当前输出

你的任务不是重写答案，而是先做 Prompt Debugging。

请按以下步骤输出：
1. 当前输出最主要的失败模式是什么
2. 这个失败模式最可能来自 Prompt 缺少哪类约束
3. 应做哪些最小必要修改
4. 为什么这些修改有效
5. 输出修复后的 Prompt

注意：
- 不要大改所有内容
- 优先做最小必要修复
- 说明修复的是哪一层控制结构
```

这个调试器的意义，在于强迫你从“坏结果”反推出“缺失结构”。这一步做熟之后，你的 prompt 质量会出现质变，因为你不再盲改。

## 九 失败模式速查表

当你看到输出不理想时，可以直接对照下面这张速查表。

```text
如果输出像复述 -> 补 问题重定义 + 认知下钻
如果输出像模板 -> 补 全局文风约束 + 禁止模板引导语
如果输出正确但空 -> 补 信息密度约束 + 关键点优先
如果输出高深但虚 -> 补 边界/验证约束 + 可执行性要求
如果输出只会赞美 -> 补 批判性指令 + 代价/局限分析
如果输出很全但不尖锐 -> 补 关键点优先 + 主问题聚焦
如果输出有洞察但难以行动 -> 补 决策收敛 + 成功标准 + 触发条件
如果输出局部很好整体不稳 -> 补 独立全局文风层 + 输出节奏约束
```

这张速查表最适合贴在你的常用工作区里。因为你真正高频需要的，不是完整理论，而是看到问题后能立即知道该补什么。

## 十 输出评估表

Prompt System 要想真正可用，就不能只靠感觉判断输出好坏。你需要一张标准评估表。下面这份评估表适合在第一版输出后直接打分。

```text
输出评估表（每项 1-5 分）

1. 是否真正回答了问题本质，而不只是复述表面需求
2. 是否识别了关键约束、边界、代价与失败模式
3. 是否有足够的信息密度，而不是正确废话
4. 是否给出了明确判断，而不是平均分配篇幅
5. 是否具备可执行性、可验证性或可决策性
6. 是否文风自然，避免明显模板腔与 AI 味
7. 是否整体稳定，而不是局部好、整体散

总分：

判断建议：
30-35：可直接使用
24-29：基本可用，但建议做一轮精修
18-23：存在明显结构缺失，建议进入 Prompt Debugging
17 以下：不要继续改答案，先重构 Prompt
```

这张表最大的价值，不是评分本身，而是把“我感觉一般”变成“到底差在哪一项”。一旦差项明确，修复路径就会清晰很多。

## 十一 日常 SOP

如果你只想记住一套最短操作流程，那么记住下面这套 SOP 就够了。

```text
SOP-1：先填任务输入卡
SOP-2：先诊断，再组装 Prompt
SOP-3：先评估输出，再决定是否调试
SOP-4：调试时只做最小必要修复
SOP-5：有效句子单独沉淀到元指令库
```

这五句几乎可以覆盖你 80% 的实际使用场景。你真正需要的不是复杂，而是稳定。

## 十二 元指令库记录模板

每当你发现一句控制语稳定有效，就不要把它埋在完整 prompt 里，而是单独抽出来记录。下面是标准记录格式。

```text
指令名称：
指令内容：
主要解决的失败模式：
适用任务：
常见收益：
可能副作用：
使用备注：
```

随着积累增加，你会逐渐形成自己的问题重定义簇、批判性簇、信息密度簇、可执行性簇和文风控制簇。这些才是你真正可以迁移到任何新任务上的资产。

## 十三 团队版使用规范

如果你准备把这份 Playbook 给团队一起用，最重要的不是让所有人学会完整理论，而是统一最低操作标准。最实用的团队规范通常只有四条。第一，复杂任务必须先有输入卡。第二，所有正式 Prompt 必须至少具备五层中的四层。第三， bad output 必须先走调试器，而不是各自自由改写。第四，每周沉淀一次有效元指令，统一进入共享库。

一旦这些规则建立起来，团队里的 prompt 质量会明显稳定，因为你不再依赖个别高手的灵感，而是依赖一套共同工作语言。

## 十四 最后原则

如果要把整份 Playbook 压缩成一句话，那就是：高质量 Prompt 不是一次写对，而是通过任务诊断、结构组装、结果评估、最小修复和经验沉淀构成的闭环系统。

当你开始按系统而不是按感觉工作时，prompt engineering 才真正从技巧变成能力。真正强的人，不是能写出一条神奇 Prompt，而是能持续、稳定、可解释地把普通需求转化成高质量输出。这份 Playbook 的全部意义，就在这里。
