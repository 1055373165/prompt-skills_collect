下面我们用一个具体的项目案例，完整模拟一次从 0 到 1 的开发过程。每个环节都标注使用了哪个工具，以及用户实际输入了什么。

### 项目案例：在线教育平台后端系统

假设用户的原始需求只有一句话：

> "我想做一个在线教育平台，支持课程管理、用户管理和视频播放。"

---

### 第一幕：需求锁定

**使用工具：需求拆解对话器**

用户实际输入：

```text
我会给你一个需求或目标。

不要直接回答，也不要直接生成 Prompt。你的任务是先用最少的问题帮我把需求锁定到可执行的精度。

你必须完成以下工作：
1. 用一句话复述你理解的核心需求
2. 识别当前描述中最大的模糊点（最多 3 个）
3. 针对每个模糊点，给出一个精准的追问（不要问开放式大问题，要给选项或范围）
4. 如果我的需求本身就足够清晰，直接告诉我"无需追问"并说明原因

规则：
- 追问总数不超过 5 个
- 每个追问必须附带 2-3 个最常见的选项，帮我快速决策
- 不要问"你的目标是什么"这类无效问题，要问具体的取舍和边界
- 追问结束后，输出一份锁定后的需求摘要

我的需求是：
我想做一个在线教育平台，支持课程管理、用户管理和视频播放。
```

AI 可能的追问输出：

```
核心需求复述：构建一个在线教育平台的后端系统，核心功能是课程 CRUD、用户认证与权限、视频内容的存储与播放。

模糊点与追问：

1. 技术选型尚未确定
   → 后端语言偏好？选项：A. Go  B. Java/Spring  C. Python/Django  D. Node.js

2. 用户规模未知，直接影响架构决策
   → 目标用户量级？选项：A. 千级（内部/小团队）B. 万级（中型平台）C. 十万级以上（需考虑分布式）

3. 视频方案未确定
   → 视频如何处理？选项：A. 接第三方云服务（如阿里云VOD）B. 自建视频转码+CDN  C. 仅存储链接，播放由前端处理
```

用户回复：

```
A. Go
B. 万级
A. 接第三方云服务
```

此时，需求已经从一句话锁定到了一个有明确边界的技术需求。

### 第二幕：架构设计

**使用工具：思维过程架构器（基础 #4）+ 架构决策记录器（新增 #2）**

用户实际输入（用思维过程架构器驱动架构思考）：

```text
在回答我的问题之前，你必须严格按照以下认知架构组织你的思考过程：

第一步：问题拆解
- 这个问题的核心矛盾是什么？
- 它涉及哪几个独立的子问题？
- 哪些子问题之间有依赖关系？

第二步：对立面检验
- 当前最主流/最直觉的答案是什么？
- 这个主流答案最可能在什么条件下失效？
- 有没有与主流答案完全相反但仍然成立的立场？

第三步：约束识别
- 回答这个问题需要哪些前提假设？
- 哪些假设是隐含的、用户没有说但你默认了的？
- 如果这些假设不成立，结论会怎么变？

第四步：综合产出
- 基于以上分析，给出你的最终回答
- 明确标注你的结论在什么范围内成立
- 如果存在无法消除的不确定性，直接说明

我的问题是：
基于以下需求，设计这个在线教育平台的后端架构：
- Go 语言
- 万级用户
- 视频接第三方云服务
- 课程管理、用户管理、视频播放三大模块
请给出分层架构、技术选型和模块划分方案。
```

AI 输出架构方案后，立即使用架构决策记录器固化关键决策：

```text
我刚刚做了一个技术决策。请你帮我把它记录成一份标准的架构决策记录。

我的决策是：
后端使用 Go + Gin 框架，数据库用 PostgreSQL，缓存用 Redis，视频服务接阿里云 VOD。
采用三层架构（Handler → Service → Repository），模块划分为 user、course、video 三个独立包。
```

**关键产出：** 一份架构方案 + 一份 ADR 记录。ADR 保存到 `DECISIONS.md`。


### 第三幕：任务拆解

**使用工具：任务拆解引擎（新增 #1）**

用户实际输入：

```text
我有一个项目目标。请你帮我把它拆解成一份可执行的结构化任务清单。

[... 任务拆解引擎完整模板 ...]

我的项目目标是：
用 Go + Gin + PostgreSQL + Redis 构建在线教育平台后端。
三大模块：用户管理（注册、登录、JWT鉴权、角色权限）、课程管理（CRUD、分类、搜索）、视频管理（接阿里云VOD、播放鉴权）。
目标用户万级，需要基本的性能优化和缓存策略。
```

AI 输出示例（摘要）：

```
阶段一：基础设施搭建（3 个任务）
  ├── 任务 1.1：项目初始化（Go module、目录结构、配置管理）
  ├── 任务 1.2：数据库搭建（PostgreSQL 连接、Migration 框架）
  └── 任务 1.3：基础中间件（日志、错误处理、CORS）

阶段二：用户模块（5 个任务）
  ├── 任务 2.1：用户数据模型与数据库表
  ├── 任务 2.2：注册接口
  ├── 任务 2.3：登录接口 + JWT 签发
  ├── 任务 2.4：JWT 鉴权中间件
  └── 任务 2.5：角色权限控制

阶段三：课程模块（4 个任务）
  ├── 任务 3.1：课程数据模型
  ├── 任务 3.2：课程 CRUD 接口
  ├── 任务 3.3：课程分类与标签
  └── 任务 3.4：课程搜索（全文检索）

阶段四：视频模块（3 个任务）
  ├── 任务 4.1：阿里云 VOD SDK 集成
  ├── 任务 4.2：视频上传与回调
  └── 任务 4.3：播放鉴权接口

阶段五：集成与优化（3 个任务）
  ├── 任务 5.1：Redis 缓存层
  ├── 任务 5.2：接口联调与集成测试
  └── 任务 5.3：性能压测与优化

关键路径：1.1 → 1.2 → 2.1 → 2.2 → 2.3 → 2.4（用户模块是其他模块的前置依赖）
```

**关键产出：** 一份包含 18 个具体任务的结构化任务清单。保存到 `PROGRESS.md`。

### 第四幕：编码实现（以一个任务为例）

**使用工具：提示词生成器（基础 #1）**

用户拿到任务清单后，从任务 2.3 开始（登录接口 + JWT 签发）。用户实际输入：

```text
我会给你一句需求、一个问题，或者一个模糊目标。

你的任务不是直接回答，而是先把它编译成一份可以直接使用的高质量 Prompt。

[... 提示词生成器完整模板 ...]

我的需求是：
用 Go + Gin 实现用户登录接口，包括密码校验和 JWT token 签发，要考虑安全性。
```

提示词生成器输出一份包含角色定义、技术约束、安全要求、代码风格要求的成品 Prompt。用户复制这个 Prompt 去获得高质量的代码实现。

### 第五幕：代码审查

**使用工具：代码审查协议（新增 #5）**

拿到 AI 生成的登录接口代码后：

```text
请你对下面这段代码做一次结构化的深层审查。

[... 代码审查协议完整模板 ...]

代码如下：
[粘贴 AI 生成的登录接口代码]

上下文信息：
- 这段代码属于用户模块（user 包）
- 它要解决的问题：处理用户登录请求，校验密码，签发 JWT
- 上下游依赖：依赖 user repository 的 FindByEmail 方法，下游是 JWT 中间件
```

审查通过后进入下一个任务。

### 第六幕：阶段验收

**使用工具：阶段验收检查器（新增 #4）**

完成阶段二（用户模块）全部 5 个任务后：

```text
我即将完成一个项目阶段，请你帮我做一次结构化的阶段验收检查。

[... 阶段验收检查器完整模板 ...]

当前阶段信息：
阶段二"用户模块"已完成，包含以下产出：
- 用户数据模型和数据库表（users 表，含 id、email、password_hash、role、created_at）
- 注册接口 POST /api/v1/register
- 登录接口 POST /api/v1/login（返回 JWT）
- JWT 鉴权中间件
- 角色权限中间件（admin / teacher / student 三种角色）
所有接口已手动测试通过。
```

验收结果示例：

```
1. 交付物完整性：✅ 5/5 交付物已产出
2. 质量门禁：⚠️ 缺少自动化测试，仅手动测试
3. 技术债务：⚠️ 密码强度校验尚未实现，当前只检查非空
4. 依赖与接口：✅ JWT 中间件接口稳定，可供后续模块使用
5. 认知对齐：✅ 与 ADR-001 架构决策一致

结论：可以进入下一阶段，但建议在阶段五补充自动化测试。
```

## 九 边缘场景完整处理方案

### 场景 A：会话中断，上下文丢失

**问题描述：** 你正在和 AI 开发项目，做到一半网络断了、token 耗尽了、或者浏览器崩了。重新开一个会话，AI 对你的项目一无所知。

**完整应对方案：**

**预防层（日常习惯）：**

在日常开发中维护两个文件，每次会话结束前更新：

1. `PROGRESS.md` — 使用进度追踪器生成，记录当前在哪、做了什么、下一步是什么
2. `DECISIONS.md` — 使用架构决策记录器生成，记录所有关键技术决策

这两个文件加上你的代码目录结构，就是完整的"项目记忆"。

**恢复层（中断后操作）：**

开新会话后，直接使用上下文恢复协议：

```text
我正在开发一个项目，但之前的 AI 会话中断了，对话历史已经丢失。

我需要你帮我恢复上下文并接续工作。

[... 上下文恢复协议完整模板 ...]

我的项目信息如下：

项目目标：用 Go 构建在线教育平台后端

当前进度（来自 PROGRESS.md）：
- 阶段一、二已完成（基础设施 + 用户模块）
- 正在进行阶段三任务 3.2（课程 CRUD 接口）
- 整体完成度约 50%

关键决策（来自 DECISIONS.md）：
- ADR-001：Go + Gin + PostgreSQL + Redis，三层架构
- ADR-002：JWT 鉴权，token 有效期 24h，refresh token 7d
- ADR-003：视频服务接阿里云 VOD，不自建转码

当前代码结构：
[粘贴 tree 命令输出或目录结构]
```

AI 基于这些信息重建上下文，列出不确定点让你确认，然后直接接续任务 3.2 继续开发。

**整个恢复过程不超过 5 分钟。**

### 场景 B：项目进度失控，不知道做到哪了

**问题描述：** 你和 AI 聊了几十轮，它一直在帮你深挖某个模块的细节，你突然意识到已经完全不知道项目整体进度了。有些功能可能做了，有些可能漏了，有些可能做到一半被打断了。

**完整应对方案：**

**紧急恢复（立即使用）：**

在当前会话中直接插入进度追踪器：

```text
停一下。我已经失去了对项目全局进度的感知。

请你根据我们这个会话中的所有对话内容，生成一份结构化的进度快照。

[... 项目进度追踪器完整模板 ...]

特别注意"被遗忘的事项检查"，回溯一下有没有提到过但还没做的任务、做到一半被打断的任务、或者因为深入某个细节而跳过的步骤。
```

**日常预防（防止再次发生）：**

建立"进度心跳"机制——每完成 3 个任务，强制触发一次进度追踪。可以在任务清单中预先插入进度检查点：

```
任务 3.1：课程数据模型
任务 3.2：课程 CRUD 接口
任务 3.3：课程分类与标签
──── 📍 进度检查点：调用进度追踪器 ────
任务 3.4：课程搜索
任务 4.1：阿里云 VOD 集成
任务 4.2：视频上传与回调
──── 📍 进度检查点：调用进度追踪器 ────
```

**深挖失控预防（从源头控制）：**

如果你发现 AI 总是在某个模块上越挖越深，可以在对话中显式插入一条控制指令：

```text
注意：当前任务是 [任务编号和名称]。
请只完成这个任务范围内的工作。
如果发现需要额外工作，标记为"待办"但不要展开。
完成后立即回到任务清单的下一项。
```

这条指令相当于给 AI 装了一个"范围锁"，防止它沿着一条链路无限制地深挖下去。

## 十 速查：什么时候用哪个工具

| 你的场景 | 应该用的工具 |
|----------|-------------|
| 有一个模糊的想法，想变成好的 Prompt | 提示词生成器 |
| 需求很复杂，一句话说不清 | 需求拆解对话器 |
| 要做架构设计或深度分析 | 思维过程架构器 |
| 项目刚启动，需要规划任务 | 任务拆解引擎 |
| 做了一个重要的技术选型 | 架构决策记录器 |
| 不知道项目做到哪了 | 项目进度追踪器 |
| 一个阶段做完了，要进入下一阶段 | 阶段验收检查器 |
| AI 写完代码了，要审查 | 代码审查协议 |
| AI 给的回答质量不行 | 输出质量自审器 |
| 会话中断了，要恢复 | 上下文恢复协议 |

## 十一 最后

这 10 个工具覆盖了大型项目 AI 辅助开发的完整生命周期。它们不是理论框架，而是可以直接复制使用的 meta-prompt 模板。

核心理念只有一条：**不要把 AI 当作一个你随便问问的聊天工具，而是把它当作一个需要明确指令、明确流程、明确检查点的协作者。你给它的结构越清晰，它返回的质量越高。**

日常使用时不需要每次都用全部 10 个工具。根据项目大小和复杂度，选择你需要的组合即可。但如果是超大型项目，建议至少使用：任务拆解引擎 + 项目进度追踪器 + 阶段验收检查器 + 上下文恢复协议。这四个是大型项目的生命线。
