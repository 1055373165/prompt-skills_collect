# 定位

**定位：把每个关键技术决策变成可追溯的结构化记录。**

大型项目中最贵的隐性成本是"忘了当初为什么这么做"。两周后你回头看代码，不知道为什么选了这个方案。三个月后新成员加入，同样的问题被重新讨论一遍。架构决策记录器的作用就是在每个关键岔路口，强制 AI 把决策过程固化下来。

**为什么不可或缺：** AI 辅助开发中，技术决策往往分散在几十轮对话里，没有任何一个地方能看到"所有关键决策的全貌"。这个工具把隐式决策变成显式记录。

# 提示词

我刚刚做了一个技术决策（或正在做一个技术决策）。请你帮我把它记录成一份标准的架构决策记录（ADR）。

你必须按照以下结构输出：

ADR 编号：[自动递增]
标题：[一句话描述这个决策]
状态：[已决定 / 待讨论 / 已废弃]
日期：[当前日期]

背景：
- 这个决策要解决的问题是什么？
- 触发这个决策的直接原因是什么？

候选方案：
- 方案 A：[描述] → 优势 / 劣势 / 风险
- 方案 B：[描述] → 优势 / 劣势 / 风险
- 方案 C：[描述]（如果有）→ 优势 / 劣势 / 风险

最终决定：
- 选择了哪个方案
- 核心理由（不超过 3 条）

代价与妥协：
- 这个决定放弃了什么？
- 在什么条件下这个决定应该被推翻？

影响范围：
- 这个决定影响了哪些模块/组件？
- 后续哪些任务需要基于这个决定来做？

规则：
- 每条 ADR 必须能独立阅读，不依赖上下文
- "核心理由"必须是技术性的，不是"感觉更好"
- "代价与妥协"不能为空，每个决策必然有取舍
- 如果我只描述了结论没给背景，你要追问背景

我的决策是：
【描述你的技术决策或正在纠结的选型问题】
