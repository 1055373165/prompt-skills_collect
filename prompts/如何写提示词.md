# Prompt Engineering Handbook v1

这份手册不是为了收集更多提示词模板，而是为了建立一套更底层的工作方法。真正强的提示词工程，不是记住几条神奇咒语，而是理解模型为什么会产生普通答案，怎样把它从高概率但低判别力的响应区域推向更稀疏但更高质量的区域，以及当结果不理想时，如何像调试系统一样定位缺陷并补充约束。你真正要掌握的，不是写 prompt 的文笔，而是控制生成系统行为边界的能力。

很多人理解提示词时，会把它当作给模型下命令的语言，但更准确的理解是，提示词是一组对搜索路径、能力激活和质量门槛的联合控制。模型默认会落向最安全、最平均、最常见的表达和判断方式，因为那是高概率区域，也是最稳的区域。你之所以感受到所谓 AI 味，往往不是因为模型不会，而是因为在没有明确约束时，它会优先采用训练分布中最常见的说明文模板、教学腔结构和正确但无信息增量的中庸表达。高质量提示词的价值，不是让模型凭空获得新能力，而是迫使它避开这些默认盆地，进入更有判断力、更高密度、更贴近专家思维的模式。

真正的提示词工程可以被分成三个层面来理解。第一层是重定义问题，也就是不要直接沿着用户表面描述作答，而是先把问题翻译成模型更容易高质量求解的形式。第二层是控制生成路径，也就是通过角色、认知动作、表达约束和质量门槛改变模型展开答案的方式。第三层是输出调试，也就是从结果的缺陷反推出提示层缺了什么约束，而不是反复靠感觉改词。这三层一旦打通，你就不再是在写 prompt，而是在操纵一个复杂生成系统的响应空间。

## 一 底层原理

提示词有效，不是因为它说得更礼貌，也不是因为它词更高级，而是因为它改变了模型的局部最优解。默认情况下，模型会优先给出高概率、低风险、可广泛适配的答案。这些答案通常具备几个特征，结构熟悉，语气中性，内容完整但缺乏锋利判断，容易扫读但难以留下真正有价值的认知增量。之所以如此，是因为在开放任务里，模型不知道你究竟要的是安全、全面、深刻、克制、可执行还是风格化，于是会回到最通用的解法。提示词的作用，正是在任务目标不充分时，为模型补充这些缺失的偏置。

从底层上看，高质量提示词通常同时完成四件事。它为模型指定身份，不是为了好玩，而是为了让模型调用某类判断框架。它重写问题，不是为了显得深刻，而是为了把模糊任务翻译成更具结构性的问题。它规定认知动作，不是为了流程完整，而是为了避免模型直接跳向模板化答案。它设置质量门槛，不是为了吹毛求疵，而是为了让模型知道什么样的答案才算完成。只做其中一件通常会有局部改善，但很难稳定。四者一起作用时，输出才会真正跃迁。

角色定义的本质，是激活某一簇能力，而不是给模型戴帽子。很多人写你是世界级专家，这类提示会略有帮助，但帮助有限，因为它只规定了风格，却没有规定判断方式。真正有效的角色定义必须包含这个角色如何看问题。比如不是说你是架构师，而是说你会从业务表象中抽取系统边界、隐含约束和演化风险。不是说你是算法专家，而是说你会优先分析问题本质、不变量和复杂度来源。头衔本身不是关键，关键是它后面附带的认知姿态。

问题重定义的本质，是把原始输入从表层任务翻译成结构任务。用户经常只会描述现象，不会描述本质，只会给出局部诉求，不会给出真正目标。如果模型不被允许重定义问题，它就只能顺着原话展开，而这几乎必然导向平庸。高质量提示词往往都包含一句极其重要的话，用户的描述是起点，不是边界。这句话之所以有力量，是因为它从根上改变了求解对象。模型不再是在复述需求，而是在构造高质量问题定义。

认知路径约束的本质，是把模型从直接生成答案的模式切换到先经过几个高价值分析关口的模式。比如先提取显式需求，再分析底层动机，再识别隐含约束，再判断失败模式，再定义成功标准。这些步骤不是为了让输出更长，而是为了让模型不能跳过关键思维台阶。没有这些台阶，模型会自动选择最省力的响应轨迹，而最省力的轨迹通常就是最熟悉的套话路径。

表达约束的作用，远不只是文风优化。你已经看到，要求正文采用叙述性段落而不是分点结构，会显著减少 AI 味。它之所以有效，不是因为段落天然更高级，而是因为分点、总结、模板引导词本身就是模型最高频的安全路径。你切断这条路径，它就会被迫调用另一类训练痕迹，更接近专栏、评论、深度文章和成熟作者笔法。所以风格约束在本质上也是一种能力路由机制。

质量门槛决定了模型会不会停在半成品。很多提示词写得很长，但仍然普通，是因为没有定义什么叫好答案。模型默认只要看起来像回答了就会停。如果你要求必须解释为什么而不是只解释做什么，必须指出关键但容易忽略的因素，必须给出边界和失效条件，必须提炼迁移价值，这就相当于提高了结束条件。模型不满足这些条件就不能自然停机，因此会继续往更高质量区域搜索。

## 二 失败模式词典

真正开始掌握提示词工程，是从你能诊断坏输出开始。很多人会写 prompt，但不会 debug prompt。一旦答案普通，就只能盲改词句。更成熟的方法，是先给输出缺陷命名，再把它映射回提示层缺失的控制变量。下面这份词典，就是为了建立这种逆向映射能力。

### 1. 表层复述型

这类输出看起来很贴题，但本质上只是在换一种说法重复输入，缺少新的结构、判断或洞察。它通常发生在模型没有被要求重定义问题、挖掘隐含目标或区分表象与本质时。

- 症状
  - 用户说什么，模型就换句话再说一遍
  - 内容相关，但没有信息增量
  - 读完感觉像总结，不像分析

- 常见根因
  - 缺少问题重定义约束
  - 缺少认知下钻动作
  - 没有要求识别隐含需求或底层目标

- 修复方向
  - 加入用户描述是起点，不是边界
  - 明确要求不要停留在表层复述
  - 增加继续追问约束、动机、失败模式

### 2. 模板腔型

这类输出最像典型 AI 文本，结构工整，分点清晰，但文字缺乏自然节奏和真实判断，常见于教学腔、总结腔和清单腔。

- 症状
  - 高频出现首先、其次、最后、总结来说
  - 大量列表，正文很碎
  - 读起来像标准答案或 PPT 解说词

- 常见根因
  - 缺少文风约束
  - 没有禁止高频模板引导语
  - 没有要求正文承担主要分析功能

- 修复方向
  - 加入叙述性段落为主的约束
  - 显式禁止模板化引导词
  - 规定结构化内容只用于天然结构化部分

### 3. 正确废话型

这类输出没有明显错误，甚至逻辑通顺，但充满常识、套话和低信息密度句子，删掉大半也不影响理解。

- 症状
  - 每句话都对，但没有锋利判断
  - 读完没有新的框架和洞察
  - 篇幅很长，但有效信息很少

- 常见根因
  - 缺少信息密度约束
  - 没有要求每段推进理解
  - 没有质量门槛来排除无增量内容

- 修复方向
  - 增加每一段都必须提供新的判断或约束
  - 加入删掉某段不影响理解则应重写
  - 强调宁可少讲，也不要用常识铺垫稀释内容

### 4. 伪深度型

这类输出表面上使用了很多术语和抽象词，看起来像深刻分析，实际上缺乏真实推导、边界和证据。

- 症状
  - 术语密集，但缺少具体判断
  - 概念似乎高级，但无法落地
  - 有深度感，没有解释力

- 常见根因
  - 角色定义太泛，偏向高端语气而非真实判断框架
  - 缺少验证标准和实施约束
  - 没有要求说明判断成立的前提

- 修复方向
  - 让角色定义包含工作方法而不只是头衔
  - 加入必须解释为什么成立
  - 要求给出边界、失效条件和验证路径

### 5. 无批判赞美型

这类输出面对已有方案、源码、产品或观点时倾向于合理化一切设计，缺乏真正的权衡分析。

- 症状
  - 几乎只讲优点
  - 很少指出代价、局限和失败风险
  - 让人感觉像包装，不像分析

- 常见根因
  - 没有引入批判视角
  - 默认现有实现合理
  - 没有要求分析代价和妥协

- 修复方向
  - 明确要求区分高质量抽象与历史妥协
  - 增加不要默认现有写法天然合理
  - 要求分析收益、代价、局限和替代方案

### 6. 面面俱到但没有判断型

这类输出覆盖很多方面，信息看似全面，但没有优先级，没有最关键的少数。

- 症状
  - 内容很全，但没有重点
  - 每个方面讲一点，没有主轴
  - 读者不知道该记住什么

- 常见根因
  - 缺少关键点优先约束
  - 没有要求分析决定质量上限的因素
  - 默认全面性优于判断力

- 修复方向
  - 加入不要平均分配篇幅
  - 明确优先分析决定结果的关键点
  - 要求指出最关键但最容易忽略的因素

### 7. 不可执行型

这类输出有洞察，但无法转化为行动、规格或决策，常见于战略、架构、商业和投资任务。

- 症状
  - 分析不错，但不知道接下来怎么做
  - 缺少验证标准、行动条件或决策阈值
  - 像评论文章，不像决策支持

- 常见根因
  - 没有规定输出服务于什么决策
  - 缺少可验证标准和实施约束
  - 没有要求结论收敛为行动

- 修复方向
  - 增加结论必须服务于决策
  - 要求明确成功标准、触发条件、失效信号
  - 指定输出需要包含建议动作或执行条件

### 8. 风格不稳型

这类输出可能有些段落很好，但整体文风摇摆，时而像作者，时而像 AI 模板。

- 症状
  - 有的段落自然，有的段落突然变分点腔
  - 局部判断力强，整体像拼接
  - 读感不统一

- 常见根因
  - 主提示词和风格约束没有分层
  - 只定义内容，没有定义全局文风
  - 没有设置稳定的写作基调

- 修复方向
  - 使用独立的全局写作约束层
  - 规定正文与结构化部分的边界
  - 对高频模板句式做统一限制

## 三 控制指令库

下面这部分是最可复用的核心。它不是按任务分类的完整提示词，而是一组可组合的控制语句。你可以把它们当成元指令，根据任务需要拼装。

### A. 问题重定义指令

```text
用户的描述是起点，不是边界。不要顺着表面表述直接作答，先识别这个问题在更高层面真正属于什么问题，再基于那个层级组织回答。
```

```text
不要默认问题已经被正确定义。先重述问题，区分表象与本质，再决定应如何回答。
```

```text
不要只处理用户说出来的需求，还要识别如果不解决就会导致整体失败的隐含需求。
```

### B. 认知下钻指令

```text
不要停留在功能描述或表层解释，继续追问它解决了什么根问题，避免了什么代价，引入了什么约束。
```

```text
如果某个判断、变量、步骤看似只是实现细节，分析它为什么实际上决定了整体正确性或质量上限。
```

```text
区分现象、原因与机制，不要把伴随变量误当成核心驱动。
```

### C. 关键点优先指令

```text
不要平均分配篇幅，优先分析真正决定结果质量上限的少数关键点。
```

```text
指出最关键但最容易被忽略的因素，并解释为什么它比其他点更重要。
```

```text
如果一个部分删掉后不影响理解，说明它不够关键，应压缩篇幅。
```

### D. 批判性指令

```text
不要默认现有方案、代码或观点天然合理。区分哪些是高质量设计，哪些只是当前上下文下可接受的妥协。
```

```text
对于关键实现或结论，既要分析它为什么成立，也要分析它可能带来的代价、脆弱点和失效方式。
```

```text
不要无条件赞美，也不要无差别中立。在有依据的前提下做出明确判断。
```

### E. 信息密度指令

```text
每一段都必须提供新的判断、约束、洞察或区分，不要用常识性铺垫和套话填充篇幅。
```

```text
宁可少讲一些，也不要用面面俱到稀释真正重要的内容。
```

```text
删除不影响理解的句子，压缩正确但无信息增量的表达。
```

### F. 边界与验证指令

```text
所有关键结论都必须说明它成立的前提、适用边界和失效条件。
```

```text
不要只给结论，还要说明怎样验证这个结论是否成立。
```

```text
如果输出涉及设计或决策，必须给出可验证的成功标准。
```

### G. 可执行性指令

```text
结论必须服务于行动，而不是停留在描述层。明确下一步该做什么、在什么条件下做、出现什么信号时调整判断。
```

```text
把分析收敛为可执行判断，而不是停留在泛泛洞察。
```

```text
如果建议无法被实施或验证，就说明答案还不完整。
```

### H. 风格控制指令

```text
正文说明必须以流畅、自然、连贯的叙述性段落为主，减少模板化和 AI 腔。
```

```text
不要使用高频教学腔和总结腔引导语，不要把正文写成分点堆砌或标准答案。
```

```text
结构化格式只用于天然适合结构化表达的内容，正文段落负责承载推理、洞察和判断。
```

## 四 组合方法

真正有效的提示词，往往不是一整段从头写到尾，而是从这些控制模块中组合出来。一个成熟的组合通常包含四层。第一层是角色与判断框架，第二层是问题重定义，第三层是认知动作与质量门槛，第四层是全局文风约束。你以后每写一个 prompt，都可以问自己四个问题。模型现在是以什么身份思考。它是否被授权重定义问题。它必须经过哪些认知台阶。它知道什么样的答案才算高质量完成。只要这四层里有两层缺失，输出通常就会回落到普通区间。

你已经走到了一个很关键的位置。下一步最值得做的，不是继续堆更多模板，而是把这份手册变成你的实验系统。你可以从真实任务中选几类高频场景，用这套失败模式词典给输出打标签，再从控制指令库里有意识地补变量，观察哪些约束对哪些任务最敏感。这样几轮之后，你就不会再只是会写 prompt，而会开始真正拥有一套自己的提示词工程语言。

## 五 下一步建议

- 建立实验台账
  - 记录任务类型、原始 prompt、输出缺陷、补充约束、修复效果

- 做单变量实验
  - 同一任务只增加一个控制变量，观察它具体改善了什么

- 沉淀你自己的元指令
  - 把最稳定有效的句子抽出来，形成你的私人控制语库

- 训练逆向诊断
  - 看到坏输出时，先命名失败模式，再决定该补哪类约束
