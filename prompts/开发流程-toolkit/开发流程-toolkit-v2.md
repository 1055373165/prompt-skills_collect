# Dev Autopilot Skill v2 — 一键开发全周期自动驾驶

## 定位

覆盖软件开发全生命周期的一键式 skill。用户输入一句需求，系统自动驱动工具链按流程执行：需求锁定 → 架构设计 → 技术验证 → 任务拆解 → 逐单元编码 → 审查 → 验收 → 进度更新。全程自动推进，仅在关键决策点暂停等待用户确认。

### v2 核心变更

| 维度 | v1 | v2 |
| --- | --- | --- |
| 需求阶段 | 4 步串联（Prompt Generator → Self-Auditor → AI → Decomposer） | 2 步直达（AI 直接展开 → Decomposer 锁定） |
| 技术验证 | 无 | 架构确定后、全量拆解前，先做技术探针 |
| 任务拆解 | 无限递归 | 最大 4 层、MDU 上限 60、熔断机制 |
| 并行策略 | 标注但不执行 | 诚实承认单线程限制，改为依赖分析 + 多会话并行指引 |
| 回溯能力 | 仅异常 C 触发 | 任意 MDU 执行中可触发显式回溯 |
| 需求变更 | 不支持 | `/dev-autopilot-change-request` 正式入口 |
| 阶段验收 | 假设 AI 能运行代码 | 区分 AI 可做（静态审查）和用户必做（运行验证） |
| 进度心跳 | 每 3 个 MDU | 每 10% 完成度或阶段切换时 |
| 恢复点 | 模糊 | 明确：总是从中断 MDU 重头开始 |
| skip 命令 | 静默跳过 | 自动标记下游阻塞的 MDU |

## 输入契约

用户只需提供一项输入：

```text
/dev-autopilot

我的需求是：
【一句需求、一个重构建议、或一个模糊目标】
```

系统接管后续所有流程。

## 输出契约

最终交付物：

1. 完整可运行的项目代码
2. `DECISIONS.md` — 所有架构决策记录
3. `PROGRESS.md` — 完整进度追踪记录（含所有阶段、任务、子任务、最小开发单元的完成状态）

## 完整执行协议

以下是 AI 必须严格遵循的分步执行协议。每一步标注了使用的工具、输入来源、输出目标和用户交互点。

**全局规则：**

- Prompt Generator 和 Output Self-Auditor 在**需求阶段不使用**，保留到**执行阶段**（MDU 编码前）发挥最大价值
- 需求阶段的瓶颈是"用户想不清楚"，只能靠追问解决，不能靠 Prompt 优化解决
- 任何阶段发现上游问题，都可以通过回溯协议回退，不必等到异常触发

---

### 第一阶段：需求理解与锁定（Steps 1-3）

**目标：** 通过 AI 直接展开 + 交互式追问，把用户的模糊需求锁定到可执行精度。

**v2 设计理由：** v1 在需求阶段用了 Prompt Generator → Self-Auditor → AI 输出 → Decomposer 四步，本质是"用 AI 优化 Prompt 再让 AI 理解优化后的 Prompt"——信息熵没有增加，但流程复杂度翻倍。需求阶段的核心瓶颈不是 Prompt 质量，而是用户自己想不清楚。去掉中间层，让 AI 直接理解用户意图，再用追问补齐缺失信息。

#### Step 1：接收原始需求

```text
输入：用户的一句话需求
动作：原样记录，不做任何修改
输出：raw_requirement
```

#### Step 2：AI 直接展开需求理解

```text
输入：raw_requirement

执行指令：
基于用户的原始需求，直接输出你对这个需求的完整理解。

你必须覆盖：
1. 你理解的核心目标（一句话）
2. 这个系统需要哪些核心能力
3. 预期的用户/调用方是谁
4. 你认为的技术边界和约束
5. 你认为用户可能没说但很重要的隐含需求
6. 你认为最容易被忽略的风险点

输出规则：
- 直接说你的理解，不要问用户"你的目标是什么"
- 如果有些信息你不确定，标注 [待确认] 而不是瞎猜
- 控制在 500 字以内

输出：ai_understanding
```

#### Step 3：Requirement Decomposer — 交互式需求锁定

```text
工具：需求拆解对话器（Requirement Decomposer）
输入：raw_requirement + ai_understanding

⚠️ 用户交互点：本步骤需要用户参与决策

执行指令：
基于用户原始需求和 AI 的理解展开，用最少的追问帮用户把需求锁定到可执行精度。

你必须完成：
1. 用一句话复述你理解的核心需求
2. 列出 AI 理解中标注 [待确认] 的所有点
3. 识别当前描述中最大的模糊点（最多 3 个）
4. 针对每个模糊点和待确认点，给出一个精准的追问（附带 2-3 个选项）
5. 如果需求已足够清晰，输出"无需追问"

追问规则：
- 总数不超过 5 个
- 每个追问必须附带选项，帮用户快速决策
- 聚焦技术取舍和边界，不问"你的目标是什么"之类的无效问题
- 追问结束后，输出一份「锁定后的最终需求摘要」

关键约束：
- 涉及技术选型、架构风格、非功能性需求的问题必须让用户决策
- 用户确认后的最终需求摘要即为后续所有流程的唯一需求基准
- 禁止在后续流程中偏离此需求摘要（除非走需求变更协议）

→ 等待用户逐一回答追问
→ 用户确认后生成最终需求摘要

输出：locked_requirement（后续所有步骤的唯一输入基准）
```

### 第二阶段：架构设计与决策固化（Steps 4-5）

**目标：** 基于锁定的需求设计架构方案，固化关键决策，识别需要技术探针验证的高风险决策。

#### Step 4：Cognitive Architecture Injector — 架构方案生成

```text
工具：思维过程架构器（Cognitive Architecture Injector）
输入：locked_requirement

执行指令：
基于以下已锁定的需求，设计完整的技术架构方案。

你必须严格按照以下认知架构组织思考：

第一步：问题拆解
- 这个系统的核心技术矛盾是什么？
- 涉及哪几个独立的技术子问题？
- 这些子问题之间的依赖关系？

第二步：对立面检验
- 最主流的架构方案是什么？
- 这个方案在什么条件下失效？
- 有没有非主流但更适合当前约束的方案？

第三步：约束识别
- 需要哪些前提假设？
- 哪些假设是隐含的？
- 如果这些假设不成立，架构怎么调整？

第四步：综合产出
输出完整架构方案，包含：
- 系统分层架构
- 模块划分与职责定义
- 技术选型（语言、框架、数据库、中间件、第三方服务）
- 模块间通信方式
- 数据模型概要
- 核心接口定义
- 非功能性设计（性能、安全、可观测性）

⚠️ 用户交互点：输出架构方案后暂停，等待用户确认或调整

输出：architecture_plan
```

#### Step 5：Architecture Decision Recorder — 固化架构决策

```text
工具：架构决策记录器（Architecture Decision Recorder）
输入：architecture_plan + 用户确认的技术选型

执行指令：
基于刚刚确定的架构方案，为每个关键技术决策生成一份 ADR。

需要记录为 ADR 的决策类型：
- 编程语言与框架选型
- 数据库选型
- 缓存策略
- 第三方服务选型
- 架构风格（单体/微服务/分层等）
- 认证鉴权方案
- 其他用户在 Step 3 中明确做出的技术取舍

每条 ADR 格式：
ADR 编号：[自动递增，从 ADR-001 开始]
标题：[一句话]
状态：已决定
日期：[当前日期]
背景：[要解决的问题 + 触发原因]
候选方案：[每个方案的优劣势]
最终决定：[选了什么 + 核心理由]
代价与妥协：[放弃了什么 + 什么条件下推翻]
推翻条件：[什么情况下应该重新评估此决策]
影响范围：[影响哪些模块 + 后续哪些任务依赖此决策]
探针验证：需要 / 不需要

探针验证判定规则：
- 需要探针验证：涉及团队未用过的技术、性能敏感的选型、第三方服务可用性依赖
- 不需要探针验证：团队熟悉的技术栈、纯架构风格选择、成熟稳定的组件

动作：将所有 ADR 写入 DECISIONS.md
输出：decisions_file + spike_candidates（标记为"需要探针验证"的 ADR 列表）
```

### 第三阶段：技术探针（Step 6）— v2 新增

**目标：** 在全量拆解前，用最小成本验证最高风险的技术决策。

**v2 设计理由：** 如果等到全量拆解并开发到第 15 个 MDU 才发现某个核心技术选型不可行，前面的工作可能全部白做。技术探针用 1-2 个最小验证，把这个风险前置消除。

#### Step 6：Technical Spike — 技术探针验证

```text
输入：spike_candidates（来自 Step 5，标记为"需要探针验证"的 ADR 列表）

如果 spike_candidates 为空：
→ 输出"所有技术决策无需探针验证，跳过本阶段"
→ 进入第四阶段

如果 spike_candidates 不为空：

对每个需要验证的 ADR：

1. 定义探针目标
   - 这个技术决策要验证什么？（一句话）
   - 验证通过的标准是什么？
   - 验证失败的回退方案是什么？

2. 设计最小验证方案
   - 编写最少量的代码来验证核心假设
   - 不做完整功能，只验证技术可行性
   - 预估代码量不超过 100 行

3. 执行验证
   - 使用 Prompt Generator 编译验证代码的实现 Prompt
   - AI 实现验证代码
   - 输出验证结果

4. 判定结果
   如果验证通过：
   → 更新 DECISIONS.md 对应 ADR 状态为"已验证（探针通过）"
   → 保留探针代码作为后续开发的参考

   如果验证失败：
   ⚠️ 用户交互点：通知用户技术探针失败
   → 分析失败原因
   → 提出替代方案（回退方案或新方案）
   → 用户确认后：
     - 更新 DECISIONS.md：将原 ADR 标记为"已推翻"，新增替代方案的 ADR
     - 如果替代方案影响架构，回溯到 Step 4 调整架构

探针阶段终止条件：所有 spike_candidates 验证通过或已更换为验证通过的替代方案
输出：validated_decisions（所有高风险决策已验证）
```

### 第四阶段：任务拆解（Steps 7-9）

**目标：** 把需求拆解到最小开发单元（MDU），标注依赖关系和执行顺序。

#### Step 7：Task Decomposition Engine — 一级任务拆解

```text
工具：任务拆解引擎（Task Decomposition Engine）
输入：locked_requirement + architecture_plan

执行指令：
基于已锁定的需求和已确定的架构方案，拆解为结构化任务清单。

拆解层级：
第一层：项目阶段（Phase）3-6 个
第二层：里程碑（Milestone）每阶段 2-4 个
第三层：具体任务（Task）每里程碑若干个

每个任务必须：
- 有一句话功能描述
- 有可验证的完成标准
- 标注依赖关系
- 标注优先级

每个阶段必须：
- 有明确的阶段交付物定义
- 有风险标注和关键路径标识

动作：将任务清单写入 PROGRESS.md
输出：task_list
```

#### Step 8：递归任务精炼 — 拆解至最小开发单元

```text
工具：需求拆解对话器（Requirement Decomposer）用于复杂任务二次拆解
输入：task_list 中的每一个任务

⚠️ 熔断机制（v2 新增）：
- 最大递归深度：4 层。超过 4 层仍有子项不满足 MDU 标准 → 立即暂停
  通知用户：这通常意味着需求范围需要重新切割，而不是继续拆。
- MDU 总数预警：拆解过程中 MDU 总数超过 60 → 立即暂停并警告用户
  可能的处理方式：
  a) 需求范围过大，应该拆成多个独立项目
  b) 拆解粒度过细，应该放宽 MDU 标准
  c) 用户确认继续
- 单任务子项上限：任何一个任务拆出的直接子项不超过 8 个
  超过说明拆解维度不对，需要先按职责或模块重新组织。

对 task_list 中的每个任务执行以下判断：

评估标准 — 该任务是否为"最小开发单元"（MDU）：
□ 功能单一，只做一件事
□ 可以在一次 AI 对话中完成编码
□ 有明确的输入输出边界
□ 可以独立测试或验证
□ 预估代码量不超过 200 行

如果以上 5 条全部满足 → 标记为最小开发单元（MDU），进入 Step 9

如果任一条不满足且未触发熔断 → 该任务需要继续拆解：

    执行指令：
    这个任务仍然太复杂，无法在一次 AI 会话中完成。
    请将它拆解为更小的子任务。

    拆解规则：
    - 每个子任务必须能用一句话描述要实现的功能
    - 每个子任务必须满足 MDU 的 5 条标准
    - 子任务之间的依赖关系必须明确
    - 如果子任务仍然不满足 MDU 标准，继续递归拆解（受熔断机制约束）

    递归终止条件：所有叶子节点都是 MDU，或触发熔断

动作：更新 PROGRESS.md，展开所有子任务和 MDU，记录最大拆解深度
输出：refined_task_tree（所有叶子节点均为 MDU）
```

#### Step 9：依赖分析与执行顺序标注

```text
输入：refined_task_tree

对每个 MDU 执行依赖分析：

判断规则：
- 如果 MDU-A 的输出是 MDU-B 的输入 → B 依赖 A
- 如果无依赖 → 标注 depends_on: none
- 如果不确定 → 默认有依赖（安全优先）

关于"并行"的诚实说明：
在单个 AI 对话窗口中，所有 MDU 事实上是串行执行的。
标注依赖关系的真正价值不是"并行执行"，而是：
1. 确定执行顺序——无依赖的 MDU 可以灵活调整顺序
2. 影响分析——当某个 MDU 失败或跳过时，知道哪些下游会被阻塞
3. 多会话并行指引——如果用户开多个窗口，无依赖的 MDU 可以在不同窗口同时推进

输出格式：为每个 MDU 标注
- depends_on: MDU-X.Y.Z / none
- blocks: MDU-A.B.C, MDU-D.E.F（被当前 MDU 阻塞的下游）

动作：更新 PROGRESS.md，为每个 MDU 标注依赖和阻塞关系
输出：execution_plan（完整的带依赖标注的 MDU 执行计划）

⚠️ 用户交互点：输出完整的执行计划后暂停，等待用户确认
  展示：
  - 总阶段数、总任务数、总 MDU 数
  - 最大拆解深度
  - 关键路径（最长依赖链）
  - 预估工作量
  用户确认后进入执行循环
```

### 第五阶段：执行循环

**目标：** 逐个完成所有 MDU 的编码、审查，并在关键节点做验收和进度更新。

#### 执行循环伪代码

```text
mdu_completed_count = 0
total_mdu_count = execution_plan.total_mdus
last_heartbeat_percent = 0

for each phase in execution_plan.phases:
    for each task in phase.tasks:
        for each subtask in task.subtasks:
            for each mdu in subtask.mdus:

                // ===== 范围锁注入 =====
                注入 Scope_Lock(mdu)

                // ===== MDU 执行流 =====
                result = MDU_Execute(mdu)

                // ===== 回溯检查 =====
                if result.needs_backtrack:
                    执行 Backtrack_Protocol(result.backtrack_target)
                    // 回溯后可能需要重新执行部分 MDU，由回溯协议决定

                // ===== 进度心跳（百分比触发）=====
                mdu_completed_count += 1
                current_percent = floor(mdu_completed_count / total_mdu_count * 100)
                if current_percent - last_heartbeat_percent >= 10:
                    输出：📍 进度心跳 | MDU {completed}/{total} | 完成度：{percent}% | 阶段：{phase.name} | 当前：{mdu.name}
                    last_heartbeat_percent = current_percent

            end for
        end for
    end for

    // ===== 阶段结束后 =====
    执行 Phase_Checkpoint(phase)
    执行 Progress_Update(phase)

end for
```

#### MDU_Execute：最小开发单元执行流

```text
函数 MDU_Execute(mdu):

    // Step A: Prompt Generator + Output Self-Auditor — 编译实现提示词
    // v2 说明：这是 Prompt Generator 在整个流程中首次登场。
    // 保留到执行阶段使用，因为此时目标明确、约束清晰，优化有着力点。
    工具：提示词生成器（Prompt Generator）+ 输出质量自审器（Output Self-Auditor）
    输入：mdu.description + 项目上下文（架构方案、已有代码结构、相关 ADR）

    执行指令：
    将下面这个最小开发单元的功能描述编译成一份高质量的代码实现 Prompt。

    功能描述：{mdu.description}

    项目上下文：
    - 架构方案：{architecture_plan 摘要}
    - 技术栈：{从 DECISIONS.md 提取}
    - 当前代码结构：{当前目录结构}
    - 相关已有代码：{mdu 依赖的上游模块代码}

    你必须在 Prompt 中包含：
    - 角色定义（该技术栈的高级工程师）
    - 技术约束（语言版本、框架版本、代码风格）
    - 安全要求（如涉及）
    - 输入输出规范
    - 错误处理要求
    - 与已有代码的集成方式
    - 不要做什么（范围锁）

    编译完成后，自动执行一轮 Self-Audit：
    - 检查信息密度、逻辑完整性、可执行性
    - 如果有 P0 问题，直接修复后输出
    - 不做多轮迭代（执行阶段的 Prompt 目标明确，一轮审计足够）

    输出：implementation_prompt

    // Step B: 实际代码实现
    输入：implementation_prompt
    动作：AI 根据 implementation_prompt 编写代码
    输出：code_changes（本次所有新增和修改的文件）

    // Step C: Code Review Protocol — 代码审查循环（最多 3 轮）
    工具：代码审查协议（Code Review Protocol）
    输入：code_changes

    review_count = 0
    review_loop:
        review_count += 1

        审查维度：
        1. 逻辑正确性（边界错误、空指针、竞态、异常路径）
        2. 架构合理性（职责单一、接口最小化、耦合度）
        3. 可维护性（可读性、隐式假设、变更成本）
        4. 性能与资源（重复计算、内存泄漏、扩展性）
        5. 安全性（输入校验、注入、越权）
        6. 与项目规范一致性（与 ADR 决策一致、与已有代码风格一致）

        输出格式：
        每个维度：✅ 无问题 / 💡 建议优化 / 🔴 必须修复

        如果存在 🔴 且 review_count < 3：
        → 给出具体修复方案
        → AI 执行修复
        → 回到 review_loop

        如果存在 🔴 且 review_count >= 3：
        → 触发异常 C（审查死循环），进入回溯分析
        → return { needs_backtrack: true, backtrack_target: 问题分析结果 }

        如果无 🔴：
        → 输出"审查通过"
        → 记录审查结果摘要

    输出：reviewed_code + review_summary

    // Step D: 更新 PROGRESS.md 中该 MDU 状态
    动作：将该 MDU 标记为 [x] 已完成

    // Step E: 上游问题检测（v2 新增）
    如果在编码过程中发现：
    - 上游 MDU 的接口定义有问题
    - 架构设计的某个假设不成立
    - 需求边界需要调整
    → 不要硬编码 workaround
    → 标记发现的问题，触发回溯协议
    → return { needs_backtrack: true, backtrack_target: 问题描述 }

    return { needs_backtrack: false }
```

#### Phase_Checkpoint：阶段验收

```text
函数 Phase_Checkpoint(phase):

    工具：阶段验收检查器（Milestone Checkpoint Validator）
    输入：phase 的所有产出 + phase 的承诺交付物定义

    执行指令：
    阶段「{phase.name}」的所有最小开发单元已完成。
    请做一次结构化的阶段验收检查。

    ── AI 可自主完成的检查 ──
    1. 交付物完整性：承诺的文件是否都已创建？代码结构是否完整？
    2. 静态代码审查：跨模块接口是否一致？命名是否统一？有无明显 copy-paste 错误？
    3. ADR 一致性：实现是否与 DECISIONS.md 中的 ADR 一致？
    4. 技术债务盘点：本阶段引入了哪些临时方案或 TODO？列出清单。
    5. 依赖就绪：下一阶段依赖的接口/数据结构是否已定义且稳定？

    ── 需要用户在本地验证的检查 ──
    6. 运行验证：请用户在本地执行以下命令，确认代码可正常运行：
       [给出具体的启动/编译/测试命令]
    7. 功能验证：请用户手动验证以下核心功能点：
       [列出本阶段的核心功能点和验证步骤]

    ⚠️ 用户交互点：等待用户反馈本地验证结果

    输出格式：
    AI 检查结果：每项 ✅ 通过 / ⚠️ 警告 / ❌ 阻断
    用户验证结果：等待用户反馈

    如果 AI 检查存在 ❌ 或用户反馈有问题：
    → 走异常 D 处理

    验收终止条件：AI 检查无 ❌ 且用户验证通过
    输出：checkpoint_result
```

```text
函数 Progress_Update(phase):

    工具：项目进度追踪器（Project Progress Tracker）
    输入：当前 PROGRESS.md + 本阶段完成情况

    执行指令：

    动作：更新 PROGRESS.md
    输出：updated_progress
```

当所有阶段通过验收后：

## 回溯协议（v2 新增）

### 显式回溯 — 任意 MDU 执行中可触发

```text
触发条件：
- MDU 执行过程中发现上游问题（MDU_Execute Step E）
- 代码审查死循环（异常 C）
- 用户主动输入 /dev-autopilot-backtrack
- 阶段验收发现架构级问题

回溯流程：

1. 问题分类
   AI 分析问题根因，归类为：
   a) 需求问题 → 回溯到 Step 3（需求锁定）
   b) 架构问题 → 回溯到 Step 4（架构设计）
   c) 技术选型问题 → 回溯到 Step 6（技术探针）
   d) 任务拆解问题 → 回溯到 Step 8（任务精炼）
   e) 单纯实现问题 → 不回溯，换思路重写当前 MDU

2. ⚠️ 用户交互点：展示回溯分析，等待用户确认回溯目标

3. 影响分析
   - 回溯后，哪些已完成的 MDU 需要重新审查或重做？
   - 哪些 ADR 需要更新？
   - PROGRESS.md 如何更新？（不删除已完成记录，标记为"因回溯需重新评估"）

4. 执行回溯
   - 回到目标步骤重新执行
   - 从目标步骤之后的所有变更按影响分析处理
   - 更新 PROGRESS.md 的变更记录

5. 回溯后恢复执行
   - 从受影响的最早 MDU 开始重新进入执行循环

关键规则：
- 每次回溯必须记录到 PROGRESS.md 的变更记录中
- 回溯不是失败，是正常的工程实践
- 如果同一个问题导致连续 2 次回溯到同一步骤，暂停并要求用户介入
```

## 需求变更协议（v2 新增）

```text
触发条件：用户输入 /dev-autopilot-change-request

变更流程：

1. 暂停当前 MDU 执行

2. 接收变更描述
   用户描述需要变更的内容

3. 影响评估
   AI 分析变更对以下内容的影响：
   - locked_requirement 中哪些条目受影响
   - DECISIONS.md 中哪些 ADR 可能需要更新
   - PROGRESS.md 中哪些 MDU 需要重做、新增或删除
   - 已完成的代码中哪些文件需要修改

4. 变更方案
   输出：
   - 受影响的 MDU 清单
   - 新增的 MDU 清单
   - 需要删除的 MDU 清单
   - 预估额外工作量

5. ⚠️ 用户交互点：用户确认变更方案

6. 执行变更
   - 更新 locked_requirement
   - 更新 DECISIONS.md（如需）
   - 更新 PROGRESS.md（标记受影响 MDU，新增变更 MDU）
   - 从受影响的最早 MDU 继续执行循环

关键规则：
- 变更记录必须写入 PROGRESS.md 的变更记录表
- 不要试图把变更"融入"现有 MDU——要么修改现有 MDU 的描述，要么新增 MDU
- 如果变更影响超过 50% 的 MDU，建议用户考虑重新启动项目
```

## 异常处理协议

### 异常 A：会话中断

```text
触发条件：用户在新会话中输入 /dev-autopilot-resume

恢复流程：
1. 读取项目根目录的 PROGRESS.md 和 DECISIONS.md
2. 读取当前代码目录结构
3. 执行上下文恢复协议（Context Recovery Protocol）：
   - 重建项目状态摘要
   - 定位当前 MDU 位置
   - 列出 3 个最不确定的点让用户确认
4. 用户确认后，从中断处的 MDU 继续执行循环

恢复点规则（v2 明确）：
- 已标记 [x] 的 MDU：不重做
- 未标记完成的当前 MDU：从头重新开始（不尝试续写半成品代码）
  原因：AI 无法可靠判断代码写到哪了，重头开始比续写半成品更安全
- 恢复后第一个动作：更新 PROGRESS.md 标记恢复点和恢复时间
```

### 异常 B：进度迷失

```text
触发条件：
- 用户主动输入 /dev-autopilot-status
- 或完成度变化超过 10% 未触发进度更新（自动检测）

恢复流程：
1. 立即暂停当前 MDU 执行
2. 调用项目进度追踪器生成完整进度快照
3. 重点执行"被遗忘的事项检查"
4. 输出进度快照给用户
5. 用户确认后继续执行

预防机制（v2 调整）：
- 进度心跳触发条件改为百分比：每 10% 完成度或每个阶段切换时
- 格式：📍 进度心跳 | MDU {X}/{Total} | 完成度：{percent}% | 阶段：{phase.name} | 当前：{mdu.name}
```

### 异常 C：代码审查死循环

```text
触发条件：同一个 MDU 的代码审查循环超过 3 次仍有 🔴 项

恢复流程：
1. 暂停审查循环
2. 输出当前所有 🔴 项的汇总
3. 分析根因，归类为：
   a) 架构问题 → 触发回溯协议，目标 Step 4
   b) 需求问题 → 触发回溯协议，目标 Step 3
   c) 技术选型问题 → 触发回溯协议，目标 Step 6
   d) 单纯实现问题 → 换一种实现思路重新编码（不回溯）

⚠️ 用户交互点：展示根因分析，用户确认处理方向
```

### 异常 D：阶段验收失败

```text
触发条件：阶段验收中 AI 检查出现 ❌ 阻断项，或用户本地验证反馈有问题

恢复流程：
1. 列出所有阻断项及其根因分析
2. 对每个阻断项生成修复方案
3. 将修复工作拆解为新的 MDU（遵循 Step 8 的 MDU 标准和熔断机制）
4. 执行这些修复 MDU（走完整的 MDU_Execute 流程）
5. 修复完成后重新执行阶段验收
6. 更新 PROGRESS.md 记录修复过程
```

## 用户交互点总表

整个流程中，AI 会在以下节点暂停等待用户输入：

| 节点 | 步骤 | 用户需要做什么 |
| --- | --- | --- |
| 需求追问 | Step 3 | 回答 3-5 个选择题，确认需求边界 |
| 架构确认 | Step 4 | 确认或调整架构方案 |
| 探针失败 | Step 6 | 确认替代方案 |
| 执行计划确认 | Step 9 | 确认 MDU 总量和关键路径后启动开发 |
| 阶段验收 | Phase_Checkpoint | 在本地运行代码并反馈验证结果 |
| 回溯确认 | 回溯协议 | 确认回溯目标和影响范围 |
| 需求变更确认 | 变更协议 | 确认变更方案和额外工作量 |
| 审查死循环 | 异常 C | 确认根因分析和处理方向 |
| 阶段验收阻断 | 异常 D | 确认修复方案 |

除以上节点外，AI 全程自动推进，不暂停。

## 范围锁机制

为防止 AI 在执行某个 MDU 时过度展开，每个 MDU 执行时自动注入以下范围锁指令：

```text
【范围锁 — 自动注入】
当前正在执行：MDU-{编号}：{描述}
属于阶段：{phase.name} → 任务：{task.name} → 子任务：{subtask.name}

严格约束：
1. 只完成本 MDU 描述范围内的工作
2. 如果发现需要额外工作，标记为"待办"写入 PROGRESS.md，但不展开
3. 不要修改不属于本 MDU 范围的文件（除非是必要的 import 或接口对接）
4. 如果发现上游问题，标记问题并触发回溯协议，不要用 workaround 绕过
5. 完成后立即报告完成状态，进入下一个 MDU
6. 不要主动优化其他模块的代码
```
