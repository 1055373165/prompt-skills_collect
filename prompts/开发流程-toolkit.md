# Dev Autopilot Skill — 一键开发全周期自动驾驶

## 定位

这是一个覆盖软件开发全生命周期的一键式 skill。用户只需输入一句需求或重构建议，系统自动驱动 10 个工具按照严格的流程编排执行，从需求编译、架构设计、任务拆解到逐单元编码、审查、验收、进度更新，全程自动化推进，仅在关键决策点暂停等待用户确认。

## 输入契约

用户只需提供一项输入：

```text
/dev-autopilot

我的需求是：
【一句需求、一个重构建议、或一个模糊目标】
```

系统接管后续所有流程。

## 输出契约

最终交付物：

1. 完整可运行的项目代码
2. `DECISIONS.md` — 所有架构决策记录
3. `PROGRESS.md` — 完整进度追踪记录（含所有阶段、任务、子任务、最小开发单元的完成状态）

## 状态文件规范

### PROGRESS.md 结构

```markdown

- 项目名称：

- 创建时间：
- 最后更新：

| 阶段 | 状态 | 任务数 | 已完成 | 完成度 |
|------|------|--------|--------|--------|

## 详细任务清单
### 阶段 X：[阶段名]
#### 任务 X.Y：[任务名]

- 子任务：
  - [x] 子任务 X.Y.1：[描述]
  - [ ] 子任务 X.Y.2：[描述]
- 最小开发单元：
  - [x] MDU-X.Y.1.1：[描述] [串行/并行]
  - [ ] MDU-X.Y.1.2：[描述] [串行/并行]

## 当前位置
- 当前阶段：
- 当前任务：
- 当前最小开发单元：
```

## 完整执行协议

以下是 AI 必须严格遵循的分步执行协议。每一步标注了使用的工具、输入来源、输出目标和用户交互点。

---

### 第一阶段：需求编译（Steps 1-3）

**目标：** 把用户的一句话需求编译成无 P0 缺陷的高质量需求提示词。

#### Step 1：接收原始需求

```
输入：用户的一句话需求
动作：原样记录，不做任何修改
输出：raw_requirement
```

#### Step 2：Prompt Generator — 需求提示词编译

```
工具：提示词生成器（Prompt Generator）
输入：raw_requirement

执行指令：
把下面这句用户需求编译成一份可以直接使用的高质量需求描述 Prompt。

你必须自动完成以下工作：
1. 识别任务类型（新功能开发 / 重构 / 优化 / bug 修复）
2. 识别用户真正想要的输出产物
3. 识别用户最在意的质量目标
4. 识别这个任务最容易出现的低质量输出
5. 自动补足最合适的角色定义、问题重定义、认知路径、质量门槛和文风约束
6. 输出一份可以直接使用的最终需求 Prompt

输出格式：
诊断摘要：
- 任务类型：
- 输出产物：
- 质量目标：
- 高风险失败模式：

编译后的需求 Prompt：
【完整成品】

输出：compiled_prompt_v1
```

#### Step 3：Output Self-Auditor — 迭代审计至无 P0

```
工具：输出质量自审器（Output Self-Auditor）
输入：compiled_prompt_v1

执行指令：
对下面这份编译后的需求 Prompt 做结构化审计。

逐项检查以下维度：
1. 信息密度：有没有正确废话、表层复述、注水段落？
2. 逻辑完整性：需求描述有没有歧义、遗漏、自相矛盾？
3. 可执行性：基于这份 Prompt 能否直接开始设计和开发？缺不缺关键前提？
4. 边界与约束：有没有说明什么不做、什么场景不支持？
5. 文风：有没有模板腔、伪深度、AI 味过重的表达？

输出格式：
逐项评分（1-5）+ 扣分原因
改进项分级：
- P0（必须修复，否则后续流程会出问题）
- P1（建议修复，提升质量）
- P2（锦上添花）

如果存在 P0 改进项：
→ 直接输出修订后的完整版本
→ 标记为 compiled_prompt_v{N+1}
→ 对修订后版本再次执行本审计流程

如果没有 P0 改进项：
→ 输出"审计通过，无 P0 改进项"
→ 当前版本即为 final_compiled_prompt

循环终止条件：审计结果无 P0 改进项
输出：final_compiled_prompt
```

---

### 第二阶段：需求锁定（Steps 4-5）

**目标：** 用编译后的需求提示词获取 AI 输出，再通过交互式追问锁定精确需求边界。

#### Step 4：获取 AI 需求理解输出

```
输入：final_compiled_prompt
动作：将 final_compiled_prompt 发送给 AI，获取 AI 对需求的完整理解和展开
输出：ai_requirement_output
```

#### Step 5：Requirement Decomposer — 交互式需求锁定

```
工具：需求拆解对话器（Requirement Decomposer）
输入：ai_requirement_output

⚠️ 用户交互点：本步骤需要用户参与决策

执行指令：
基于下面 AI 对需求的理解输出，用最少的追问帮用户把需求锁定到可执行精度。

你必须完成：
1. 用一句话复述你理解的核心需求
2. 识别当前描述中最大的模糊点（最多 3 个）
3. 针对每个模糊点，给出一个精准的追问（附带 2-3 个选项）
4. 如果需求已足够清晰，输出"无需追问"

追问规则：
- 总数不超过 5 个
- 每个追问必须附带选项，帮用户快速决策
- 聚焦技术取舍和边界，不问"你的目标是什么"之类的无效问题
- 追问结束后，输出一份「锁定后的最终需求摘要」

关键约束：
- 涉及技术选型、架构风格、非功能性需求的问题必须让用户决策
- 用户确认后的最终需求摘要即为后续所有流程的唯一需求基准
- 禁止在后续流程中偏离此需求摘要

→ 等待用户逐一回答追问
→ 用户确认后生成最终需求摘要

输出：locked_requirement（后续所有步骤的唯一输入基准）
```

### 第三阶段：架构设计（Steps 6-7）

**目标：** 基于锁定的需求设计架构方案，固化关键决策。

#### Step 6：Cognitive Architecture Injector — 架构方案生成

```
工具：思维过程架构器（Cognitive Architecture Injector）
输入：locked_requirement

执行指令：
基于以下已锁定的需求，设计完整的技术架构方案。

你必须严格按照以下认知架构组织思考：

第一步：问题拆解
- 这个系统的核心技术矛盾是什么？
- 涉及哪几个独立的技术子问题？
- 这些子问题之间的依赖关系？

第二步：对立面检验
- 最主流的架构方案是什么？
- 这个方案在什么条件下失效？
- 有没有非主流但更适合当前约束的方案？

第三步：约束识别
- 需要哪些前提假设？
- 哪些假设是隐含的？
- 如果这些假设不成立，架构怎么调整？

第四步：综合产出
输出完整架构方案，包含：
- 系统分层架构
- 模块划分与职责定义
- 技术选型（语言、框架、数据库、中间件、第三方服务）
- 模块间通信方式
- 数据模型概要
- 核心接口定义
- 非功能性设计（性能、安全、可观测性）

⚠️ 用户交互点：输出架构方案后暂停，等待用户确认或调整

输出：architecture_plan
```

#### Step 7：Architecture Decision Recorder — 固化架构决策

```
工具：架构决策记录器（Architecture Decision Recorder）
输入：architecture_plan + 用户确认的技术选型

执行指令：
基于刚刚确定的架构方案，为每个关键技术决策生成一份 ADR。

需要记录为 ADR 的决策类型：
- 编程语言与框架选型
- 数据库选型
- 缓存策略
- 第三方服务选型
- 架构风格（单体/微服务/分层等）
- 认证鉴权方案
- 其他用户在 Step 5 中明确做出的技术取舍

每条 ADR 格式：
ADR 编号：[自动递增，从 ADR-001 开始]
标题：[一句话]
状态：已决定
日期：[当前日期]
背景：[要解决的问题 + 触发原因]
候选方案：[每个方案的优劣势]
最终决定：[选了什么 + 核心理由]
代价与妥协：[放弃了什么 + 什么条件下推翻]
影响范围：[影响哪些模块 + 后续哪些任务依赖此决策]

动作：将所有 ADR 写入 DECISIONS.md
输出：decisions_file（DECISIONS.md 已创建）
```

### 第四阶段：任务拆解（Steps 8-10）

**目标：** 把需求拆解到最小开发单元（MDU），标注串并行关系。

#### Step 8：Task Decomposition Engine — 一级任务拆解

```
工具：任务拆解引擎（Task Decomposition Engine）
输入：locked_requirement + architecture_plan

执行指令：
基于已锁定的需求和已确定的架构方案，拆解为结构化任务清单。

拆解层级：
第一层：项目阶段（Phase）3-6 个
第二层：里程碑（Milestone）每阶段 2-4 个
第三层：具体任务（Task）每里程碑若干个

每个任务必须：
- 有一句话功能描述
- 有可验证的完成标准
- 标注依赖关系
- 标注优先级

每个阶段必须：
- 有明确的阶段交付物定义
- 有风险标注和关键路径标识

动作：将任务清单写入 PROGRESS.md
输出：task_list
```

#### Step 9：递归任务精炼 — 拆解至最小开发单元

```
工具：需求拆解对话器（Requirement Decomposer）用于复杂任务二次拆解
输入：task_list 中的每一个任务

对 task_list 中的每个任务执行以下判断：

评估标准 — 该任务是否为"最小开发单元"（MDU）：
□ 功能单一，只做一件事
□ 可以在一次 AI 对话中完成编码
□ 有明确的输入输出边界
□ 可以独立测试或验证
□ 预估代码量不超过 200 行

如果以上 5 条全部满足 → 标记为最小开发单元（MDU），进入 Step 10

如果任一条不满足 → 该任务需要继续拆解：

    执行指令：
    这个任务仍然太复杂，无法在一次 AI 会话中完成。
    请将它拆解为更小的子任务。

    拆解规则：
    - 每个子任务必须能用一句话描述要实现的功能
    - 每个子任务必须满足 MDU 的 5 条标准
    - 子任务之间的依赖关系必须明确
    - 如果子任务仍然不满足 MDU 标准，继续递归拆解

    递归终止条件：所有叶子节点都是 MDU

动作：更新 PROGRESS.md，展开所有子任务和 MDU
输出：refined_task_tree（所有叶子节点均为 MDU）
```

#### Step 10：标注串并行关系

```
输入：refined_task_tree

对每个 MDU 执行依赖分析：

判断规则：
- 如果 MDU-A 的输出是 MDU-B 的输入 → 串行（B 依赖 A）
- 如果 MDU-A 和 MDU-B 无数据依赖、无接口依赖、不修改相同文件 → 并行
- 如果 MDU-A 和 MDU-B 修改同一个文件但不同函数/方法 → 标记为"条件并行"（需谨慎）
- 如果不确定 → 默认串行（安全优先）

输出格式：为每个 MDU 标注
- [串行] 依赖：MDU-X.Y.Z
- [并行] 无依赖
- [条件并行] 与 MDU-X.Y.Z 修改同文件

动作：更新 PROGRESS.md，为每个 MDU 标注执行方式
输出：execution_plan（完整的带串并行标注的 MDU 执行计划）

⚠️ 用户交互点：输出完整的执行计划后暂停，等待用户确认
  - 展示：总阶段数、总任务数、总 MDU 数、预估工作量
  - 用户确认后进入执行循环
```

### 第五阶段：执行循环

**目标：** 逐个完成所有 MDU 的编码、审查，并在每个阶段结束后做验收和进度更新。

#### 执行循环伪代码

```
for each phase in execution_plan.phases:
    for each task in phase.tasks:
        for each subtask in task.subtasks:
            for each mdu in subtask.mdus:

                // ===== MDU 执行流 =====
                执行 MDU_Execute(mdu)

            end for
        end for
    end for

    // ===== 阶段结束后 =====
    执行 Phase_Checkpoint(phase)
    执行 Progress_Update(phase)

end for
```

#### MDU_Execute：最小开发单元执行流

```
函数 MDU_Execute(mdu):

    // Step A: Prompt Generator — 编译实现提示词
    工具：提示词生成器（Prompt Generator）
    输入：mdu.description + 项目上下文（架构方案、已有代码结构、相关 ADR）

    执行指令：
    将下面这个最小开发单元的功能描述编译成一份高质量的代码实现 Prompt。

    功能描述：{mdu.description}

    项目上下文：
    - 架构方案：{architecture_plan 摘要}
    - 技术栈：{从 DECISIONS.md 提取}
    - 当前代码结构：{当前目录结构}
    - 相关已有代码：{mdu 依赖的上游模块代码}

    你必须在 Prompt 中包含：
    - 角色定义（该技术栈的高级工程师）
    - 技术约束（语言版本、框架版本、代码风格）
    - 安全要求（如涉及）
    - 输入输出规范
    - 错误处理要求
    - 与已有代码的集成方式
    - 不要做什么（范围锁）

    输出：implementation_prompt

    // Step B: 实际代码实现
    输入：implementation_prompt
    动作：AI 根据 implementation_prompt 编写代码
    输出：code_changes（本次所有新增和修改的文件）

    // Step C: Code Review Protocol — 代码审查循环
    工具：代码审查协议（Code Review Protocol）
    输入：code_changes

    review_loop:
        执行指令：
        对本次 MDU 开发的所有改动文件做结构化深层审查。

        审查维度：
        1. 逻辑正确性（边界错误、空指针、竞态、异常路径）
        2. 架构合理性（职责单一、接口最小化、耦合度）
        3. 可维护性（可读性、隐式假设、变更成本）
        4. 性能与资源（重复计算、内存泄漏、扩展性）
        5. 安全性（输入校验、注入、越权）
        6. 与项目规范一致性（与 ADR 记录的决策一致、与已有代码风格一致）

        输出格式：
        每个维度：✅ 无问题 / 💡 建议优化 / 🔴 必须修复

        如果存在 🔴 必须修复项：
        → 给出具体修复方案
        → AI 执行修复
        → 将修复后的代码重新提交审查
        → 回到 review_loop

        如果没有 🔴 必须修复项：
        → 输出"审查通过"
        → 记录审查结果摘要

    审查循环终止条件：无 🔴 项
    输出：reviewed_code + review_summary

    // Step D: 更新 PROGRESS.md 中该 MDU 状态
    动作：将该 MDU 标记为 [x] 已完成
    输出：progress_updated
```

#### Phase_Checkpoint：阶段验收

```
函数 Phase_Checkpoint(phase):

    工具：阶段验收检查器（Milestone Checkpoint Validator）
    输入：phase 的所有产出 + phase 的承诺交付物定义

    执行指令：
    阶段「{phase.name}」的所有最小开发单元已完成。
    请做一次结构化的阶段验收检查。

    逐项检查：
    1. 交付物完整性：承诺的交付物是否全部产出且可用？
    2. 质量门禁：代码能否正常运行？核心功能有无测试覆盖？有无已知 bug？
    3. 技术债务：本阶段引入了哪些临时方案？是否阻碍下一阶段？
    4. 依赖与接口：下一阶段依赖的接口/数据结构是否稳定？
    5. 认知对齐：实现是否与 DECISIONS.md 中的 ADR 一致？

    输出格式：
    每个维度：✅ 通过 / ⚠️ 警告 / ❌ 阻断
    最终结论：可以进入下一阶段 / 需要先处理 N 个阻断项

    如果存在 ❌ 阻断项：
    → 列出具体阻断原因和修复方案
    → 执行修复
    → 重新验收
    → 循环直至无阻断项

    验收循环终止条件：无 ❌ 阻断项
    输出：checkpoint_result
```

#### Progress_Update：进度更新

```
函数 Progress_Update(phase):

    工具：项目进度追踪器（Project Progress Tracker）
    输入：当前 PROGRESS.md + 本阶段完成情况

    执行指令：
    阶段「{phase.name}」已通过验收。请更新进度快照。

    必须输出：
    项目总览：总阶段数 / 已完成 / 进行中
    当前位置：刚完成的阶段 / 下一个阶段
    完成度：基于 MDU 数量计算（已完成 MDU / 总 MDU）
    进度健康度：是否偏离计划 / 最大风险
    被遗忘的事项检查：有无遗漏、打断、跳过的任务
    接下来应该做什么：下一阶段的第一个 MDU

    动作：更新 PROGRESS.md
    输出：updated_progress
```

### 第六阶段：全局收尾

```
当所有阶段通过验收后：

1. 最终全局审查
   工具：输出质量自审器（Output Self-Auditor）
   输入：完整项目代码 + DECISIONS.md + PROGRESS.md

   审查维度：
   - 所有 ADR 决策是否在代码中得到体现？
   - 模块间接口是否一致？
   - 有无跨模块的重复代码或不一致命名？
   - 整体代码质量是否均匀（没有某个模块明显比其他差）？

2. 最终进度快照
   将 PROGRESS.md 更新为最终版本，所有 MDU 标记为完成，完成度 100%

3. 项目交付摘要
   输出一份简短的项目完成报告：
   - 项目名称和目标
   - 总耗时（阶段数 × 任务数 × MDU 数）
   - 关键架构决策数量
   - 最终代码结构
   - 已知的技术债务和后续建议
```

## 异常处理协议

### 异常 A：会话中断

```
触发条件：用户在新会话中输入 /dev-autopilot-resume

恢复流程：
1. 读取项目根目录的 PROGRESS.md 和 DECISIONS.md
2. 读取当前代码目录结构
3. 执行上下文恢复协议（Context Recovery Protocol）：
   - 重建项目状态摘要
   - 定位当前 MDU 位置
   - 列出 3 个最不确定的点让用户确认
4. 用户确认后，从中断处的 MDU 继续执行循环

关键规则：
- 不要重做已完成的 MDU
- 如果中断发生在某个 MDU 的代码审查循环中，从该 MDU 重新开始
- 恢复后的第一个动作必须是更新 PROGRESS.md 标记恢复点
```

### 异常 B：进度迷失

```
触发条件：
- 用户主动输入 /dev-autopilot-status
- 或 AI 检测到连续 5 个 MDU 后未触发进度更新

恢复流程：
1. 立即暂停当前 MDU 执行
2. 调用项目进度追踪器生成完整进度快照
3. 重点执行"被遗忘的事项检查"
4. 输出进度快照给用户
5. 用户确认后继续执行

预防机制（内置于执行循环）：
- 每完成 3 个 MDU，自动触发一次轻量级进度检查（不生成完整快照，只输出当前位置和完成度百分比）
- 格式：📍 进度心跳 | MDU {X}/{Total} 已完成 | 当前：{mdu.name} | 完成度：{X/Total * 100}%
```

### 异常 C：代码审查死循环

```
触发条件：同一个 MDU 的代码审查循环超过 3 次仍有 🔴 项

恢复流程：
1. 暂停审查循环
2. 输出当前所有 🔴 项的汇总
3. 分析是否是需求本身有问题、架构设计缺陷、还是实现层面的问题
4. 如果是架构问题 → 回溯到 Step 7，考虑是否需要新增或修改 ADR
5. 如果是需求问题 → 回溯到 Step 5，与用户重新确认该 MDU 的需求边界
6. 如果是实现问题 → 换一种实现思路重新编码

⚠️ 用户交互点：任何回溯操作都需要用户确认
```

### 异常 D：阶段验收失败

```
触发条件：阶段验收中出现 ❌ 阻断项

恢复流程：
1. 列出所有阻断项及其根因分析
2. 对每个阻断项生成修复方案
3. 将修复工作拆解为新的 MDU（遵循 Step 9 的 MDU 标准）
4. 执行这些修复 MDU（走完整的 MDU_Execute 流程）
5. 修复完成后重新执行阶段验收
6. 更新 PROGRESS.md 记录修复过程
```

## 用户交互点总表

整个流程中，AI 会在以下节点暂停等待用户输入：

| 节点 | 步骤 | 用户需要做什么 |
|------|------|----------------|
| 需求追问 | Step 5 | 回答 3-5 个选择题，确认需求边界 |
| 架构确认 | Step 6 | 确认或调整架构方案 |
| 执行计划确认 | Step 10 | 确认任务清单和 MDU 总量后启动开发 |
| 审查死循环 | 异常 C | 决定回溯方向 |
| 阶段验收阻断 | 异常 D | 确认修复方案 |

除以上节点外，AI 全程自动推进，不暂停。

## 范围锁机制

为防止 AI 在执行某个 MDU 时过度展开，每个 MDU 执行时自动注入以下范围锁指令：

```text
【范围锁 — 自动注入】
当前正在执行：MDU-{编号}：{描述}
属于阶段：{phase.name} → 任务：{task.name} → 子任务：{subtask.name}

严格约束：
1. 只完成本 MDU 描述范围内的工作
2. 如果发现需要额外工作，标记为"待办"写入 PROGRESS.md，但不展开
3. 不要修改不属于本 MDU 范围的文件（除非是必要的 import 或接口对接）
4. 完成后立即报告完成状态，进入下一个 MDU
5. 不要主动优化其他模块的代码
```
